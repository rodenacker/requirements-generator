Normalisation notes

  • [AI-SUGGESTED] Multi-value attribute — user.Roles: in strict 3NF, a multi-value enum list normalises to a separate role entity plus a user-role junction table. Acceptable as denormalisation here because the role enum is small and fixed ({ Importer, Approver }) and admin-managed; promote to a junction if role catalogues grow or carry per-role metadata.
  • [AI-SUGGESTED] Read-side denormalisation — transaction.FileName: explicitly denormalised from file-log.FileName per §7 notes ("denormalised from FileLog for table display"). Acceptable for table-rendering performance; needs a write-time consistency rule if the parent FileName ever changes.
  • [AI-SUGGESTED] Snapshot at write time — file-log.SettingName: explicit snapshot of file-setting.Name captured at upload per §7. Intentional immutability (the file-log keeps the setting name it was uploaded under, even if the underlying FileSetting is later renamed); no consistency rule needed.
  • [AI-SUGGESTED] Audit-string vs FK — LastChangedUser (on user, file-log, transaction) is stored as text rather than a User FK reference. Stable identity is preserved only as long as the captured string (likely email) does not change. Consider promoting to a reference typed FK to user in a physical model where audit-query joins matter.
  • [AI-SUGGESTED] Relationship-to-FK gap — The user → uploads → file-log, user → approves → transaction, and user → rejects → transaction relationships in §2.2 are not expressed as explicit FK columns in §7; they are captured implicitly via the LastChangedUser audit string. Consider adding UploadedByUserId (file-log) and an ActedByUserId/ApprovedByUserId/RejectedByUserId column (transaction) in a physical model for join efficiency and stable identity.
