UC-01

Authenticate

USER-GOAL
● Importer or Approver

An authenticated user gains access to their role-specific landing page.

Stakeholders
  • Importer or Approver — reach the role-specific working surface.
  • Audit team — every authentication attempt recorded with timestamp and identity.
  • Security team — failed attempts and lockout policy enforced consistently.
Preconditions
  • User account exists with bcrypt-hashed credentials (§6.6.1; §7 User).
  • User provides their registered email and password (§5 Authentication trigger).
Success guarantees
  • Session cookie issued with HttpOnly, Secure, SameSite=Strict attributes (§6.6.1).
  • User is routed to the role-specific landing page (§5 Authentication step 4).
  • Authentication event recorded in the audit trail (§6.6.4).
Minimal guarantees
  • Error message does not reveal which credential field was incorrect (BR-05).
  • Failed attempt is counted toward the lockout policy (§6.6.1).
  • Failed attempt is recorded in the audit trail (§6.6.4).
Trigger
The user submits credentials on the login page (§5 Authentication).

Main success scenario (flow-from-task-flows)

  1. 1The usersubmits credentials.
  2. 2The Systemvalidates the credentials against the hashed credential store.
  3. 3The Systemsets the HttpOnly, Secure, SameSite=Strict session cookie.
  4. 4The Systemroutes the user to the role-specific landing page.

Extensions

  1. 1aexception
    1. The System rejects the attempt with a generic error message that does not reveal which credential field was incorrect.
    2. The System increments the failed-attempt counter for the account.
    3. After five consecutive failures, the System places the account in a 15-minute cooldown.
  2. 2aexception
    1. The account is in a cooldown window; the System rejects authentication with the same generic message.
UC-02

Upload transaction file

USER-GOAL
● Importer

The Importer submits a transaction file so the system can parse it into a File Log and downstream Transactions.

Stakeholders
  • Importer — visible acceptance feedback to trust the daily ingestion cycle.
  • Audit team — uploads recorded with LastChangedUser / LastChangedDate (§6.6.4).
  • Upstream operations — visible if files fail to load (§3 Importer fears).
Preconditions
  • Importer is authenticated and holds X access on File Upload (§6.5 RBAC).
  • FileSettingId references an existing FileSetting (§7 File Log; §9 FileSetting).
Success guarantees
  • A File Log record is created with FileName, SettingId, SettingName, and an initial state (§7 File Log; §6.1).
  • The Importer is informed of acceptance and the File Log's current status is rendered (§5 File Upload step 5).
  • Upload audited via LastChangedUser / LastChangedDate (§6.3; §6.6.4).
Minimal guarantees
  • On failure, no File Log row is left in a Completed state (§5 File Upload exception path).
  • Failed upload attempt is recorded in the audit trail (§6.6.4).
Trigger
The Importer chooses a transaction file to upload (§5 File Upload trigger).

Main success scenario (flow-from-task-flows)

  1. 1The Importerchooses a transaction file to upload.
  2. 2The Importerprovides FileSettingId, FileSettingName, and FileName.
  3. 3The Importersubmits the upload.
  4. 4The Systemcreates a File Log and parses the file into Transactions.
  5. 5The Systemreports upload success and renders the File Log's current status.

Extensions

  1. 2aexception
    1. The System keeps the submit control disabled until FileSettingId, FileSettingName, and FileName are all provided.
  2. 4aexception
    1. File parsing fails; the System marks the File Log as Failed and does not create transactions in a Completed state.
    2. The System reports the failure to the Importer.
UC-03

Monitor uploaded file processing status

USER-GOAL
● Importer or Approver

The Importer (or any authenticated user) confirms each uploaded file reached a non-failed state and can drill into a file's transactions when needed.

Stakeholders
  • Importer — needs to act if any file is in a Failed state (§3 Importer).
  • Audit team — file states retained for audit (§6.6.4).
Preconditions
  • User is authenticated (§6.5 RBAC).
Success guarantees
  • The System lists every uploaded file with File Name, Process Date, Record Count, and Status (§5 File Log Overview step 2; §6.1).
  • Each file's current state is visible and labelled (§7 File Log enum).
Minimal guarantees
  • Empty state names "File Logs" and offers the upload CTA only to Importers (§5 File Log Overview exception path).
  • Failed files remain visible with their Failed status surfaced prominently (BR-07).
Trigger
The user opens the File Logs dashboard (§5 File Log Overview trigger).

Main success scenario (flow-from-task-flows)

  1. 1The useropens the File Logs dashboard.
  2. 2The Systemdisplays each File Log with File Name, Process Date, Record Count, and Status.
  3. 3The userreviews each row's status.

Extensions

  1. 2aalt
    1. No files have been uploaded yet; the System renders the empty state naming "File Logs" and offers the upload CTA only to Importers.
  2. 2bexception
    1. One or more files are in Failed state; the System surfaces the Failed status prominently and inhibits drill-down into transactions until retry succeeds.

  • Pagination control offers 5 / 10 / 20 / 50 rows-per-page with default 20 (§6.4).
Minimal guarantees

  • No PII is rendered in error toasts or browser console (§6.6.4).
Trigger

Main success scenario (flow-from-task-flows)

  2. 2The Systemrenders Reference, Date, Account, Amount, Currency, and Status columns.
  3. 3The userbrowses the table, scrolling, paging, or sorting.

Extensions

    1. No transactions exist; the System renders the empty state naming "Transactions".
  2. 3aalt
    1. The user narrows the view via UC-05 Filter and search transactions.
UC-06

Approve a transaction

USER-GOAL
● Approver

The Approver moves a single Imported transaction to Approved status with a confirmation step that prevents accidental approvals.

Stakeholders
  • Approver — personally accountable for every approval (§3 Approver).
  • Audit team — approve event recorded with LastChangedUser / LastChangedDate (§6.3; §6.6.4).
  • Downstream consumers — approved transactions flow downstream.
Preconditions
  • Approver is authenticated and holds A†BR-01 access on Approve Transaction (§6.5 RBAC).
  • Transaction is in Imported status (BR-01; §2.3 Transaction invariant).
Success guarantees
  • The transaction status transitions Imported → Approved (BR-01; §2.3).
  • The new status is reflected in the transaction table without a manual refresh (BR-06).
  • Approve event audited with LastChangedUser / LastChangedDate (§6.3; §6.6.4).
Minimal guarantees
  • Default focus on the confirmation modal is on Cancel, not on the destructive primary (§6.4).
  • On failure, no partial status change occurs (data integrity preserved). derived-from-constraints
Trigger
The Approver decides to approve an Imported transaction (§5 Approve Transaction).

Main success scenario (flow-from-task-flows)

  1. 1The Approverchooses an Imported transaction to approve.
  2. 2The Approverinitiates approval.
  3. 3The Systemasks the Approver to confirm and identifies the affected transaction.
  4. 4The Approverconfirms the approval.
  5. 5The Systemtransitions the transaction status to Approved and records the change in the audit trail.
  6. 6The Systemrefreshes the transaction table to reflect the new status.

Extensions

  1. 1aexception
    1. The transaction is not in Imported status; the System hides the Approve action on the row.
  2. 4aalt
    1. The Approver cancels at the confirmation modal; the System dismisses the modal with no status change.
  3. 5aexception
    1. Step-up authentication is required; the System prompts for MFA before recording the approval.
UC-07

Reject a transaction with a note

USER-GOAL
● Approver

The Approver moves a single Imported transaction to Rejected status, capturing a mandatory rationale that lands in the audit record.

Stakeholders
  • Approver — personally accountable; the note is the audit artefact (§3 Approver).
  • Audit team — rejection rationale retained for 7 years (§6.6.4).
  • Compliance review — note is the audit-trail explanation.
Preconditions
  • Approver is authenticated and holds A†BR-02 access on Reject Transaction (§6.5 RBAC).
  • Transaction is in Imported status (BR-01; §2.3 Transaction invariant).
Success guarantees
  • The transaction status transitions Imported → Rejected (BR-01; §2.3).
  • The non-empty user note is persisted against the Transaction (BR-02; §7 Transaction UserNote).
  • The new status is reflected in the transaction table without a manual refresh (BR-06).
  • Reject event audited with LastChangedUser / LastChangedDate (§6.3; §6.6.4).
Minimal guarantees
  • Submit remains disabled while the note is empty; inline validation flags the field as required (§5 Reject Transaction exception; BR-02).
  • On failure, no partial status change occurs (data integrity preserved). derived-from-constraints
Trigger
The Approver decides to reject an Imported transaction (§5 Reject Transaction).

Main success scenario (flow-from-task-flows)

  1. 1The Approverchooses an Imported transaction to reject.
  2. 2The Approverinitiates rejection.
  3. 3The Systemasks the Approver for a mandatory rejection note.
  4. 4The Approverprovides the rejection note.
  5. 5The Approversubmits the rejection.
  6. 6The Systemtransitions the transaction status to Rejected, persists the note, and records the change in the audit trail.
  7. 7The Systemrefreshes the transaction table to reflect the new status.

Extensions

  1. 1aexception
    1. The transaction is not in Imported status; the System hides the Reject action on the row.
  2. 4aexception
    1. The Approver leaves the note empty; the System keeps Submit disabled and surfaces the note field as required.
  3. 5aalt
    1. The Approver cancels the reject form; the System dismisses the form with no status change.
  4. 6aexception
    1. Step-up authentication is required; the System prompts for MFA before recording the rejection.

  • Audit team — exports retained for traceability (§6.6.4).

Minimal guarantees

UC-05

Filter and search transactions

SUBFUNCTION
● Importer or Approver

The authenticated user narrows a list view to the relevant subset of records.

Stakeholders
  • Authenticated user — relies on filters to find work-of-interest quickly.
Preconditions
  • A list view (File Log Overview, Transaction Table) is open (§5 Search & Filtering).
Success guarantees
  • Active filters are reflected as chips; Clear-all is offered (§5 Search & Filtering decision points).
  • Filters apply across Status, File (FileLogId), Date range, Amount range, and free text on Reference / Account (§5; §6.1).
Minimal guarantees
  • Zero-results state names the active filters and suppresses the create CTA (§5 Search & Filtering exception path; §6.4).
Trigger
The user begins narrowing the list (§5 Search & Filtering).

Main success scenario (flow-from-task-flows)

  3. 3The Systemapplies the filters and updates the list.

Extensions

  1. 3aalt
