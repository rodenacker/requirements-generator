UC-10

Run a transaction approval cycle

SUMMARY
● Importer

The Importer initiates a transaction approval cycle that ends when the file's transactions reach final disposition and the approved set is available to downstream consumers.

Stakeholders
  • Importer — files reach final disposition (§1 Application context).
  • Approver — all imported transactions are reviewed (§1).
  • Audit team — every approve/reject/upload/login/logout event is logged (§6.6.4).
  • POPIA Regulator — 7-year audit retention (§6.6.4).
  • Downstream consumers — receive the exported approved set (§1).
Preconditions
  • The Importer has access to upload. [§6.5 RBAC]
  • The Approver has access to review / approve / reject / export. [§6.5 RBAC]
  • A FileSetting referenced by the upload exists. [§7 File Log]
Success guarantees
  • Every transaction produced by the uploaded File Log has reached final disposition (Approved or Rejected). derived-from-pains
  • The approved-and-exported set is available to downstream consumers. derived-from-pains
Minimal guarantees
  • The audit trail records every step of the cycle. [§6.6.4]
  • Data integrity is preserved across the multi-step pipeline (no partial commits). derived-from-risks
Trigger
An Importer has a new transaction file to upload. derived-trigger

Main success scenario (flow-derived)

  1. 1The Importerauthenticates (Include UC-01).
  2. 2The Importeruploads a transaction file (Include UC-02).
  3. 3The Approverauthenticates (Include UC-01).
  4. 4The Approverreviews the imported transactions (Include UC-04).
  5. 5The Approverapproves or rejects each transaction (Include UC-05 or UC-06).
  6. 6The Approverexports the approved set (Include UC-07).

Extensions

  1. 2aexception
    1. The Importer resolves the upload failure (Include UC-02 retry path) before the cycle can continue. derived-from-risks

Sea-level UCs: a single sitting; the actor walks away with the goal achieved or explicitly abandoned. The default for ambiguous classification.

UC-01

Log in to the system

USER-GOAL
● User

An authenticated user logs in so they can access their role-specific landing page and begin work.

Stakeholders
  • User — successful, secure authentication into the role-specific landing page.
  • Security team — account-protection policies enforced (lockout, bcrypt, session cookie attributes per §6.6.1).
Preconditions
  • The user holds valid credentials in the system. [§6.1, §7 User]
  • The user is at the login page. [§5 Flow: Authentication]
Success guarantees
  • The user holds a valid session cookie (HttpOnly, Secure, SameSite=Strict). §6.6.1
  • The user is routed to their role-specific landing page. §5 Flow: Authentication
Minimal guarantees
  • On failed authentication, the error message does not reveal which credential field was incorrect. [BR-05]
  • On repeated failure, the account is locked per the lockout policy (5 attempts → 15-minute cooldown). [§6.6.1]
  • The login attempt is recorded in the audit trail. [§6.6.4]
Trigger
The user opens the login page. [§5 Flow: Authentication]

Main success scenario (flow-from-task-flows)

  1. 1The Usersubmits their email and password.
  2. 2The Systemvalidates the credentials against the bcrypt-hashed credential store.
  3. 3The Systemissues a session cookie with HttpOnly, Secure, SameSite=Strict attributes.
  4. 4The Systemroutes the User to the role-specific landing page.

Extensions

  1. 1aexception
    1. The System returns a generic error response that does not reveal which credential field was incorrect. [BR-05]
    2. The User remains on the login page. [§5 Flow: Authentication exception]
  2. 1bexception
    1. The System rejects the login attempt for the 15-minute cooldown after 5 failed attempts. [§6.6.1 Account lockout]
    2. The System informs the User with a generic error message. [BR-05 + §6.6.1]
UC-02

Upload a transaction file

USER-GOAL
● Importer

The Importer uploads a transaction file with required metadata so the system can ingest it into a File Log and parse its records.

Stakeholders
  • Importer — upload acceptance and parsing feedback (§3 Importer driving forces).
  • Approver — well-formed transactions arrive in the review queue.
  • Upstream operations — visible accountability for failed files (§3 Importer "Stakes").
Preconditions
  • The user is authenticated as an Importer. [§6.5 RBAC, BR-04]
  • The user holds an active session. [§6.6.1]
  • A FileSetting referenced by the upload exists. [§7 File Log: SettingId references FileSetting.Id]
Success guarantees
  • A File Log row is created with FileSettingId, FileSettingName, FileName, and the original filename. §5 Flow: File Upload + §6.1
  • The file's processing lifecycle is exposed via LastExecutedActivityName / CurrentStatus on the File Log. §6.1, §7 File Log
  • Upload feedback is rendered to the Importer after submission. §5 Flow: File Upload
Minimal guarantees
  • On failure, no File Log row is created in a Completed state. [§5 Flow: File Upload exception]
  • On failure, the file is not parsed into Transactions. [§5 Flow: File Upload exception]
  • The upload event is recorded in the audit trail. [§6.6.4]
Trigger
The Importer selects a file to upload. [§5 Flow: File Upload]

Main success scenario (flow-from-task-flows)

  1. 1The Importerprovides the required metadata (FileSettingId, FileSettingName, FileName) for the file.
  2. 2The Importersubmits the upload.
  3. 3The Systemcreates a File Log with the supplied metadata.
  4. 4The Systembegins parsing the file into Transactions.
  5. 5The Systemrenders upload feedback to the Importer.

Extensions

  1. 1aexception
    1. The System prevents upload submission until all required metadata is supplied. [§5 Flow: File Upload decision points]
  2. 4aexception
    1. The System records the File Log in a Failed state. [§7 File Log; §5 Flow: File Upload exception]
    2. The System surfaces the failure to the Importer. [§5 Flow: File Upload exception]
    3. The System inhibits drill-down from the file's summary view into transactions until retry succeeds. [BR-07]
UC-03

Review file processing status

USER-GOAL
● Importer

The Importer reviews the file log dashboard to confirm uploaded files have reached a completed state and to surface any failures.

Stakeholders
  • Importer — file-level processing confirmation (§3 Importer driving forces).
  • Upstream operations — knowing any failed files require attention (§3 Importer fears).
Preconditions
  • The user is authenticated. [§6.5 RBAC, §6.6.1]
  • At least one File Log exists OR the empty-state copy is rendered. [§5 Flow: File Log Overview]
Success guarantees
  • The Importer sees the list of uploaded files with columns File Name, Process Date, Record Count, and Status. §6.1, §5 Flow: File Log Overview
  • The Importer can drill into a file's summary view to see total records and counts by status (Imported / Approved / Rejected). §5 Flow: File Summary View, §6.1
Minimal guarantees
  • Empty list shows the empty-state message naming the entity ("No file logs yet"). [§5 Flow: File Log Overview exception, §6.4]
  • A failed file's state is surfaced prominently and drill-down into transactions is inhibited until retry succeeds. [BR-07]
Trigger
The Importer opens the File Logs dashboard. [§5 Flow: File Log Overview]

Main success scenario (flow-from-task-flows)

  1. 1The Systemrenders the list of uploaded files with columns File Name, Process Date, Record Count, and Status.
  2. 2The Importerlocates the most recent uploads and inspects each file's status.
  3. 3The Importerdrills into a file's summary view.
  4. 4The Systemrenders the File Summary View with total records and a count by status (Imported / Approved / Rejected).

Extensions

  1. 1aalt
    1. The System renders the empty-state message naming the entity ("No file logs yet"). [§5 Flow: File Log Overview exception]
    2. The System offers the upload CTA to Importers only. [§5 Flow: File Log Overview "Role-conditional behaviour"]
  2. 4aalt
    1. The System surfaces the file's current state prominently in the summary view. [BR-07]
    2. The System inhibits drill-down into transactions until retry succeeds. [BR-07]
UC-04

Review imported transactions

USER-GOAL
● Approver

The Approver reviews the queue of Imported transactions to decide which ones to approve and which to reject.

Stakeholders
  • Approver — queue clarity and decision confidence (§3 Approver driving forces).
  • Audit team — every reviewed transaction is recorded with LastChangedUser / LastChangedDate (§6.6.4).
  • Downstream consumers — only approved transactions flow downstream (§1 Application context).
Preconditions
  • The user is authenticated as an Approver. [§6.5 RBAC]
  • The transaction table is reachable from navigation or from a file log row. [§5 Flow: Transaction Table]
Success guarantees
  • The Approver sees the transaction table with columns Reference, Date, Account, Amount, Currency, and Status. §6.1, §5 Flow: Transaction Table
  • The Approver can browse Imported transactions and apply filters via UC-08. §5 Flow: Transaction Table, §6.1
  • Approve / Reject row actions are visible only for rows in Imported status. BR-01
Minimal guarantees
  • Empty filter results show active filter chips and a Clear-all action. [§5 Flow: Search & Filtering exception, §6.4]
  • Empty data ("no data") is distinguished from "no results from a filter". [§5 Flow: Transaction Table exception]
Trigger
The Approver opens the transaction table from navigation or from a file log row. [§5 Flow: Transaction Table]

Main success scenario (flow-from-task-flows)

  1. 1The Systemrenders the transaction table with columns Reference, Date, Account, Amount, Currency, and Status.
  2. 2The Approverscans the transactions in the Imported state.
  3. 3The Approverapplies filters to narrow the queue (UC-08).
  4. 4The Approverdetermines whether each row warrants an approve or reject decision.
  5. 5The Systemkeeps the row-level Approve / Reject actions available for rows in Imported status only.

Extensions

  1. 1aalt
    1. The System renders the zero-results state naming the active filters. [§5 Flow: Search & Filtering exception, §6.4]
    2. The System offers Clear-all instead of the create CTA. [§6.4]
  2. 1balt
    1. The System renders the empty-state message ("No transactions yet"). [§6.4]
    2. The System does not offer a create CTA (transactions are produced by upload, not created in-app). [§5 Flow: Transaction Table exception]
UC-05

Approve a transaction

USER-GOAL
● Approver

The Approver approves a single Imported transaction with a confirmation step so it cannot be approved by accident.

Stakeholders
  • Approver — personal accountability for the approval (§3 Approver "Stakes").
  • Audit team — approval recorded with LastChangedUser / LastChangedDate (§6.6.4).
  • Downstream consumers — only correctly approved records propagate (§1).
Preconditions
  • The user is authenticated as an Approver. [§6.5 RBAC, BR-03]
  • The target transaction's Status is Imported. [BR-01]
  • The Approver has located the transaction in the table. [§5 Flow: Approve Transaction]
Success guarantees
  • The transaction's Status transitions from Imported to Approved. §5 Flow: Approve Transaction
  • The new status is reflected in the transaction table without a manual refresh. BR-06
  • LastChangedUser and LastChangedDate are recorded on the transaction. §6.3, §6.6.4
  • The approval event is recorded in the audit trail. §6.6.4
Minimal guarantees
  • On confirmation cancel, the transaction's Status remains Imported. [§5 Flow: Approve Transaction step 3]
  • On service failure, the transaction's Status is not partially updated (no partial commit). derived-from-risks
  • The attempt is recorded in the audit trail. [§6.6.4]
Trigger
The Approver selects a target transaction and initiates approval. [§5 Flow: Approve Transaction]

Main success scenario (flow-from-task-flows)

  1. 1The Approverinitiates the Approve action on the target Imported transaction.
  2. 2The Systemrenders a confirmation modal naming the affected transaction.
  3. 3The Approverconfirms the approval.
  4. 4The Systemtransitions the transaction's Status from Imported to Approved.
  5. 5The Systemrecords LastChangedUser and LastChangedDate on the transaction.
  6. 6The Systemrefreshes the transaction table so the new status is visible without a manual refresh.

Extensions

  1. 1aexception
    1. The System hides the Approve action on the row when the transaction is not in Imported status. [BR-01]
  2. 3aalt
    1. The System dismisses the confirmation modal. [§5 Flow: Approve Transaction step 3]
    2. The transaction's Status remains Imported. [§2.3 Transaction invariant]
  3. 4aexception
    1. The System surfaces the failure to the Approver. derived-from-risks
    2. The transaction's Status remains Imported (no partial commit). derived-from-risks
UC-06

Reject a transaction

USER-GOAL
● Approver

The Approver rejects a single Imported transaction and records a mandatory note so the rejection rationale is captured for audit.

Stakeholders
  • Approver — defensible rejection rationale (§3 Approver driving forces).
  • Audit team — mandatory note captured for audit (BR-02, §6.6.4).
  • POPIA Regulator — 7-year retention of the rejection rationale (§6.6.4).
Preconditions
  • The user is authenticated as an Approver. [§6.5 RBAC, BR-03]
  • The target transaction's Status is Imported. [BR-01]
  • The Approver has located the transaction in the table. [§5 Flow: Reject Transaction]
Success guarantees
  • The transaction's Status transitions from Imported to Rejected. §5 Flow: Reject Transaction
  • A non-empty UserNote is recorded against the transaction. BR-02, §7 Transaction.UserNote
  • LastChangedUser and LastChangedDate are recorded on the transaction. §6.3, §6.6.4
  • The new status is reflected in the transaction table without a manual refresh. BR-06
  • The rejection event is recorded in the audit trail. §6.6.4
Minimal guarantees
  • If the note field is empty, the submit control remains disabled and inline validation flags the field as required. [§5 Flow: Reject Transaction exception, BR-02]
  • On submit failure, the transaction's Status is not partially updated (no partial commit). derived-from-risks
  • The attempt is recorded in the audit trail. [§6.6.4]
Trigger
The Approver selects a target transaction and initiates rejection. [§5 Flow: Reject Transaction]

Main success scenario (flow-from-task-flows)

  1. 1The Approverinitiates the Reject action on the target Imported transaction.
  2. 2The Systempresents a mandatory note field.
  3. 3The Approverprovides a non-empty note explaining the rejection.
  4. 4The Approversubmits the rejection.
  5. 5The Systemtransitions the transaction's Status from Imported to Rejected.
  6. 6The Systemrecords the UserNote, LastChangedUser, and LastChangedDate on the transaction.
  7. 7The Systemrefreshes the transaction table so the new status is visible without a manual refresh.

Extensions

  1. 1aexception
    1. The System hides the Reject action on the row when the transaction is not in Imported status. [BR-01]
  2. 3aexception
    1. The System keeps the submit control disabled while the note field is empty. [§5 Flow: Reject Transaction exception, BR-02]
    2. The System surfaces inline validation flagging the note as required. [§5 Flow: Reject Transaction exception]
  3. 5aexception
    1. The System surfaces the failure to the Approver. derived-from-risks
    2. The transaction's Status remains Imported (no partial commit). derived-from-risks
UC-07

Export filtered transactions

USER-GOAL
● Approver

The Approver exports the currently filtered transaction set to CSV for downstream consumption.

Stakeholders
  • Approver — faithful export of the currently filtered set (§3 Approver driving forces).
  • Downstream consumers — receive the exported CSV (§1 Application context).
Preconditions
  • The user is authenticated as an Approver. [§6.5 RBAC]
  • A transaction table view is active. [§5 Flow: Export Transactions]
Success guarantees
  • A CSV file is generated and downloaded. §5 Flow: Export Transactions, §6.1
  • The CSV reflects the active filter set, not the full table. §5 Flow: Export Transactions decision points, §6.1
Minimal guarantees
  • An empty filtered dataset still produces a header-only CSV; the user is informed via a toast. [§5 Flow: Export Transactions exception]
Trigger
The Approver initiates the Export action from the transaction table. [§5 Flow: Export Transactions]

Main success scenario (flow-from-task-flows)

  1. 1The Approveroptionally applies filters to narrow the dataset (UC-08).
  2. 2The Approverinitiates the Export action.
  3. 3The Systemgenerates a CSV reflecting the active filter set.
  4. 4The Systemdelivers the CSV file to the Approver's browser for download.

Extensions

  1. 3aalt
    1. The System generates a header-only CSV when the filtered dataset is empty. [§5 Flow: Export Transactions exception]
    2. The System informs the Approver via a toast. [§5 Flow: Export Transactions exception, §6.4]
UC-09

Log out of the system

USER-GOAL
● User

An authenticated user invalidates their session so the cookie is cleared and access ends.

Stakeholders
  • User — secure session termination at end of work.
  • Security team — session cookie is cleared (§6.6.1).
Preconditions
  • The user holds an active session. [§6.6.1]
Success guarantees
  • The session cookie is cleared via POST /v1/auth/logout (Max-Age=0). §6.6.1, §6.1
  • The user is no longer authenticated; subsequent requests require re-authentication. §6.6.1
  • The logout event is recorded in the audit trail. §6.6.4
Minimal guarantees
  • On logout request failure, the session is not partially invalidated. derived-from-risks
  • The logout attempt is recorded in the audit trail. [§6.6.4]
Trigger
The user decides to end their session. derived-trigger

Main success scenario (flow-derived)

  1. 1The Userinitiates the logout action.
  2. 2The Systeminvalidates the session cookie via Max-Age=0 on POST /v1/auth/logout.
  3. 3The Systemredirects the User to the login page.

Extensions

  1. 2aexception
    1. The System informs the User of the logout failure. derived-from-risks
    2. The session cookie remains set until retry. derived-from-risks

UC-08

Filter a transaction list

SUBFUNCTION
● User

A user narrows a transaction (or file log) list to a workable subset using Status, File, Date range, Amount range, or free-text filters.

Stakeholders
  • User — fast scan of a workable subset (§3 Approver driving forces, §3 Importer driving forces).
Preconditions
  • The user is authenticated. [§6.5 RBAC]
  • A transaction (or file log) list is rendered. [§5 Flow: Search & Filtering]
Success guarantees
  • The list updates to show only rows matching the active filters. §5 Flow: Search & Filtering, §6.1
  • Filter chips reflect every active filter and offer Clear-all. §5 Flow: Search & Filtering decision points, §6.4
Minimal guarantees
  • Zero-results state names the active filters and offers Clear-all rather than the create CTA. [§5 Flow: Search & Filtering exception, §6.4]
Trigger
The User applies filters or enters search text on a transaction or file log list. [§5 Flow: Search & Filtering]

Main success scenario (flow-from-task-flows)

  1. 1The Userchooses and applies one or more filter values (Status, File, Date range, Amount range, or text search).
  2. 2The Systemupdates the list to show only matching rows.
  3. 3The Systemrenders filter chips reflecting the active filters and offers Clear-all.

Extensions

  1. 2aalt
    1. The System renders the zero-results state with active filter chips and a Clear-all action. [§5 Flow: Search & Filtering exception, §6.4]
