UC-01

Authenticate

USER-GOAL
● User

The User submits credentials to gain access to the system in their authorised role so they can use the Importer or Approver functionality.

Stakeholders
  • The User — gain access to their authorised role.
  • Audit team — every login event recorded with user identity and timestamp (§6.6.4).
  • Security / compliance — credentials handled per security policy (bcrypt storage; generic error; lockout per §6.6.1).
Preconditions
  • The User has a registered account with credentials in the system. §7 User
  • The User is not currently locked out. §6.6.1
Success guarantees
  • A session cookie (HttpOnly, Secure, SameSite=Strict) is set on the User's browser. §6.6.1
  • The UI routes to the role-specific landing page (Upload for Importer; Transaction Table for Approver). §5 Auth
  • The login event is recorded with LastChangedUser and LastChangedDate. §6.6.4
Minimal guarantees
  • The error response does not reveal which credential field was incorrect. BR-05
  • The lockout counter is updated on failed attempts. §6.6.1
  • Data integrity is preserved (the session is either fully established or not at all; no partial commits). derived-from-constraints
Trigger
The User reaches the login page and enters email + password. §5 Auth

Main success scenario (flow-from-task-flows)

  1. 1 The User opens the login page.
  2. 2 The User submits credentials.
  3. 3 The System validates the credentials and sets the session cookie (HttpOnly, Secure, SameSite=Strict).
  4. 4 The System routes the User to the role-specific landing page.

Extensions

  1. 2a exception
    1. The System returns a generic error message that does not reveal which credential field was incorrect (BR-05).
    2. The System updates the lockout counter for the User account.
    3. The flow returns to step 1.
  2. 2b exception
    1. The System rejects the credentials for the duration of the 15-minute cooldown after 5 failed attempts (§6.6.1).
    2. The flow returns to step 1 once the cooldown expires.
UC-02

Upload a transaction file

USER-GOAL
● Importer

The Importer submits a transaction file with the required metadata so the system can ingest it into a new File Log for downstream processing.

Stakeholders
  • The Importer — file accepted and parsed (§3 Importer wants).
  • Approvers (downstream) — parsed transactions available for review.
  • Audit team — every upload event recorded with LastChangedUser and LastChangedDate (§6.6.4).
  • Upstream operations — visible failure if files fail to load (§3 Importer stakes).
Preconditions
  • The User is authenticated as Importer. §6.5 RBAC
  • The Importer has provided FileSettingId, FileSettingName, and FileName. §5 File Upload
Success guarantees
  • A File Log row is created with FileName, RecordCount, SettingId, SettingName, and CurrentStatus. §7 / §6.1
  • The upload event is recorded with LastChangedUser and LastChangedDate. §6.6.4 / §6.3
  • The Importer receives immediate feedback on whether the file was accepted. §4.2 / §6.1
Minimal guarantees
  • On failure, no File Log row is created in a Completed state. §5 File Upload
  • Data integrity is preserved (no partial commits). derived-from-constraints
  • The failure attempt is recorded in the audit trail. derived-from-risks
Trigger
The Importer has a transaction file ready to ingest. §5 File Upload

Main success scenario (flow-from-task-flows)

  1. 1 The Importer chooses the file to upload (drag-and-drop supported).
  2. 2 The Importer provides FileSettingId, FileSettingName, and FileName.
  3. 3 The Importer submits the upload.
  4. 4 The System creates a File Log row and begins processing.
  5. 5 The System displays the upload result (success or failure) to the Importer.

Extensions

  1. 2a exception
    1. The System inhibits the submit control until required metadata (FileSettingId, FileSettingName, FileName) is provided.
    2. The flow returns to step 2.
  2. 4a exception
    1. The System marks the File Log row as Failed (LastExecutedActivityName = Failed).
    2. The System displays the failure to the Importer.
    3. The System inhibits drill-down into transactions until retry succeeds (BR-07).

UC-05

Browse the transaction table

USER-GOAL
● Approver

The Approver opens the transaction list so they can scan imported transactions and decide which to approve and which to reject.

Stakeholders
  • The Approver — fast scan of the queue with clear filters (§3 Approver wants).
  • Downstream consumers — only approved transactions propagate.
Preconditions
  • The User is authenticated. §6.5 RBAC
Success guarantees
  • The Approver has scanned the transaction list with Reference, Date, Account, Amount, Currency, and Status columns. §6.1 / §5
  • The list reflects the active filter set (or the unfiltered set if no filters applied). §5 Search & Filtering
Minimal guarantees
  • Row-level Approve / Reject actions are hidden if the User is not authorised (Importer). BR-03
  • The table data is read-only during the scan (no mutations without explicit action). derived-from-constraints
Trigger
The Approver opens the transaction table from navigation or from a file log row. §5 Transaction Table

Main success scenario (flow-from-task-flows)

  1. 1 The Approver opens the transaction table.
  2. 2 The System renders the transactions with Reference, Date, Account, Amount, Currency, and Status columns.
  3. 3 The Approver scans the transactions and, optionally, narrows the view via filtering and search (UC-09).
  4. 4 The Approver decides which transactions to approve or reject.

Extensions

  1. 2a alt
    1. The System displays a zero-results state that names the active filters and offers a Clear-all action (suppressing the create CTA) when a filter yields no rows (§5 Transaction Table / §5 Search & Filtering Exception paths).
  2. 2b alt
    1. The System displays an empty state that names the entity ("No transactions yet") when no transactions exist in the system (§5 Transaction Table Exception paths).
UC-06

Approve a transaction

USER-GOAL
● Approver

The Approver records a positive decision against a single Imported transaction so it moves to the Approved status and is consumable downstream.

Stakeholders
  • The Approver — defensible decision (§3 Approver stakes).
  • Audit team — every approval recorded with LastChangedUser and LastChangedDate (§6.6.4).
  • Downstream consumers — approved transactions feed downstream.
Preconditions
  • The User is authenticated as Approver. §6.5 / BR-03
  • The selected transaction's status is Imported. BR-01
  • The Approver has completed step-up authentication for this session. derived-from-constraints
Success guarantees
  • The transaction's status changes to Approved. §2.3 / §6.1
  • The transaction table reflects the new status without a manual refresh. BR-06
  • The approval is recorded with LastChangedUser and LastChangedDate. §6.6.4
Minimal guarantees
  • The transaction is not modified if the Approver cancels at the confirmation modal (default focus on Cancel). §5 / §6.4
  • The attempt is recorded in the audit trail. §6.6.4
  • Data integrity is preserved (no partial commits). derived-from-constraints
Trigger
The Approver selects a transaction and invokes the Approve action. §5 Approve Transaction

Main success scenario (flow-from-task-flows)

  1. 1 The Approver selects the transaction to approve.
  2. 2 The Approver invokes the Approve action on the selected transaction.
  3. 3 The Approver confirms the action in the confirmation modal naming the affected transaction.
  4. 4 The System updates the transaction's status to Approved and reflects the change in the transaction table without a manual refresh.

Extensions

  1. 1a exception
    1. The System hides the Approve action when the selected transaction's status is not Imported (BR-01 / §2.3 Transaction invariant).
  2. 3a alt
    1. The System dismisses the confirmation modal without changing the transaction's status when the Approver cancels (§6.4 default focus on Cancel).
    2. The flow returns to step 1.

Reject a transaction with mandatory note

USER-GOAL
● Approver

The Approver records a negative decision with a mandatory rationale against a single Imported transaction so the Rejected status and the rationale are captured for audit.

Stakeholders
  • The Approver — defensible rejection with rationale (§3 Approver stakes).
  • Audit team — rejection recorded with user note (§6.6.4).
  • Originating department — rationale visible for follow-up.
Preconditions
  • The User is authenticated as Approver. §6.5 / BR-03
  • The selected transaction's status is Imported. BR-01
  • The Approver has completed step-up authentication for this session. derived-from-constraints
Success guarantees
  • The transaction's status changes to Rejected. §2.3 / §6.1
  • The mandatory user note is captured against the transaction. BR-02
  • The transaction table reflects the new status without a manual refresh. BR-06
  • The rejection is recorded with LastChangedUser and LastChangedDate. §6.6.4
Minimal guarantees
  • If the note field is empty, the submit control remains disabled. BR-02 / §5
  • The attempt is recorded in the audit trail. §6.6.4
  • Data integrity is preserved (no partial commits). derived-from-constraints
Trigger
The Approver selects a transaction and invokes the Reject action. §5 Reject Transaction

Main success scenario (flow-from-task-flows)

  1. 1 The Approver selects the transaction to reject.
  2. 2 The Approver invokes the Reject action on the selected transaction.
  3. 3 The Approver records the mandatory rejection note.
  4. 4 The Approver submits the rejection.
  5. 5 The System updates the transaction's status to Rejected, stores the user note, and reflects the change in the transaction table without a manual refresh.

Extensions

  1. 1a exception
    1. The System hides the Reject action when the selected transaction's status is not Imported (BR-01 / §2.3 Transaction invariant).
  2. 3a exception
    1. The System keeps the submit control disabled while the note field is empty (BR-02 / §5 Exception paths).
    2. The System flags the note field with inline validation indicating it is required.
    3. The flow returns to step 3.
UC-08

Export filtered transactions to CSV

USER-GOAL
● Approver

The Approver downloads the currently filtered transaction view as a CSV file so it can be handed to downstream consumers.

Stakeholders
  • The Approver — faithfulness to current filter (§4.1 G-03 quality signals).
  • Downstream consumers — predictable CSV format (§4.1 G-03).
Preconditions
  • The User is authenticated as Approver. §6.5 RBAC
  • The transaction table is open with an active filter set (or none). §5 Export
Success guarantees
  • A CSV file is generated reflecting the currently filtered dataset. §5 / §6.1
  • The CSV is delivered to the Approver as a download. §5 Export
Minimal guarantees
  • An empty filtered dataset still produces a header-only CSV; the Approver is informed via a toast. §5 Export Exception paths
  • Data integrity is preserved (no partial CSV writes). derived-from-constraints
Trigger
The Approver invokes the Export action from the transaction table. §5 Export Transactions

Main success scenario (flow-from-task-flows)

  1. 1 The Approver applies filters to the transaction table (optional).
  2. 2 The Approver invokes the Export action.
  3. 3 The System generates a CSV reflecting the active filter set and delivers it to the Approver as a download.

Extensions

  1. 1a alt
    1. The Approver skips filtering; the flow continues with the unfiltered dataset (§5 Export — filters optional).
  2. 3a alt
    1. The System still produces a header-only CSV when the filtered dataset is empty (§5 Export Exception paths).
    2. The System informs the Approver via a toast that the dataset was empty.
