Business rules

id target rule provenance
BR-01 transaction When a transaction's status is not Imported, then the Approve and Reject row actions must be hidden. §2.3
BR-02 transaction When an Approver rejects a transaction, then a non-empty user note must be supplied. §2.3
BR-03 user When the authenticated role is Importer, then the Approve and Reject actions must be hidden across the UI. §6.5
BR-04 user When the authenticated role is Approver, then the File Upload entry point and route must be hidden. §6.5
BR-05 user When login fails, then the error response and the UI message must not reveal which credential field was incorrect. §6.2
BR-06 transaction When a transaction's status changes (Approved or Rejected), then the new status must be reflected in the transaction table without a manual refresh. §2.3
BR-07 file-log When a file log is in Failed state, then the file summary view must surface the failure prominently and inhibit drill-down into transactions until retry succeeds. §2.3
BR-08 file-log A File Log cannot be approved or rejected as a whole; only individual transactions may be acted on. Status actions affect transaction status only, not file structure. §2.3
BR-09 audit-event Every approve, reject, upload, login, and logout event must be recorded as an audit event with actor and timestamp, and retained for 7 years. §6.6.4

Normalisation notes

  • Intentional denormalisation — transaction.file_name: duplicates file-log.file_name (joinable via transaction.file_log_id). Per §7 explicitly intended for table-display performance; not a 3NF defect.
  • Snapshot field — file-log.setting_name: captures file-setting.name at upload time and is not refreshed if the referenced file-setting is later renamed. Per §7 an intentional audit-snapshot.
  • [AI-SUGGESTED] Possible overlap — file-log.current_status and file-log.last_executed_activity_name: both express lifecycle state. §7 describes current_status as the "operational status complement to LastExecutedActivityName" but the semantic boundary is not defined — consider whether both fields are needed or one can be derived.
