Requirements Generator — System Flowchart

How the nine slash commands fit together. Base pipelines produce the core deliverables — requirements, PRD, design system, low-fi wireframes, and finally a hi-fi clickable prototype. Optional pipelines lens the material at two moments — before /requirements (input reviews & analyses, PRD) and after it (requirement reviews & analyses). The design / build layer runs last: /wireframe explores low-fi directions, then /prototype builds the chosen direction into a clickable React app. Click any block in the flowchart to expand its description below.

§2Reviews vs Analyses — purpose at a glance

Both optional pipeline families lens the same source material (raw input/ or merged requirements.md), but they produce categorically different outputs and demand different consultant actions.

Reviews

Critique — what is wrong with the source

Purpose
Surface defects, gaps, ambiguities, contradictions, completeness holes, and risks in the source material.
Output shape
A punch-list of findings — never an auto-patch. The artefact is something the consultant reads and acts on, not a positive contract for downstream tools.
Consultant action
Manual edits to input/ files (for /review-inputs) or to requirements.md (for /review-requirement). Some findings also generate follow-up elicitation questions for the client.
Pick when
You suspect the source is incomplete, ambiguous, or weak, and you want a structured list of specifically what to fix before drafting or designing further.
Examples
adversarial · completeness-review · ambiguity-review · first-principles · ten-ba-questions · ten-ux-questions · user-stories · gap-analysis

Analyses

Transform — make implicit structure explicit

Purpose
Derive a structural model (objects, jobs, journeys, tasks, themes, state lifecycles, …) from the source so downstream design phases can key off an explicit contract instead of improvising.
Output shape
A positive artefact — object map, theme map, task tree, journey card, state diagram, opportunity-solution tree — that another pipeline can consume as input.
Consultant action
Copy the artefact into input/ for re-ingestion by the next /requirements pass (for /analyse-inputs), or for /analyse-requirement — select it as an input to /wireframe.
Pick when
You want to enrich the source with a structural lens (motivation, object model, process flow, sentiment journey) before drafting or designing — and you want that lens to flow forward as a positive contract.
Examples
ooux · jtbd · journey-mapping · task-analysis · opportunity-solution-trees · state-diagram · data-model · use-cases · thematic-analysis · affinity-mapping · mvp-slicing · crud-coverage · glossary · user-goal-analysis · business-context-definition

§3How outputs flow back into inputs

Pipeline outputs are not auto-piped. The consultant manually shuttles relevant pieces between runs — every cross-pipeline transfer is an explicit decision with a paper trail.

The four handoff patterns

FromWhat the consultant does with the output
/generate-prd Copy strategic content (success metrics, MVP phasing, hypotheses) into input/ before running /requirements.
/analyse-inputs · any methodology Copy the derived artefact (theme map, task tree, OOUX object model, JTBD job map …) from analyse-inputs/<METHOD>/ into input/ before running /requirements. Most analysers ship a re-ingestion block (YAML/JSON + Mermaid) that survives markitdown HTML → MD conversion.
/review-inputs · any methodology Read the punch-list and manually edit the source files in input/ — add missing context, resolve ambiguity, surface stakeholder voices. Then re-run /requirements against the improved corpus.
/review-requirement · any methodology Read the punch-list and manually edit requirements/requirements.md to address defects — or re-run /requirements with corrective input. The review never auto-patches the spec.
Select relevant analyses (OOUX object map, user-journeys, task-flows, trade-off-dimension matrix) as supporting inputs when you run /wireframe (Stage 1b — completed analyses are read sidecar-first) or /prototype (basis B of its A–E input selection), so the produced screens and prototype reflect the lensed structure. They also remain reference reading you use to set scope and trade-off-dimension positions.

/review-inputs Optional · pre-requirements · 4 methodologies
↑ Back to flowchart

Critique the raw input/ corpus — missing context, ambiguity, contradiction, stakeholder-voice gaps. Output is a punch-list you act on by hand-editing files in input/ before running /requirements.

adversarial

BMAD six-dimension critique
Inputs
Every file in input/ via source-manifest.json. Treats the corpus as the stakeholder voice, not evidence-of-elicitation.
When to use
Before /requirements when you suspect the input set is uneven — coverage holes, mixed first- / second-hand voices, sources contradicting each other.
Delivers
review-inputs/ADVERSARIAL/adversarial-review.html — six-dimension findings (stakeholder coverage, workflow coverage, ambiguity, cross-source conflict, quantitative signal, scope signal) with corpus-handling recommendations.
How to use
Hand-edit the source files in input/ to address each finding (add context, reconcile conflict, label second-hand voices), then re-run /requirements.

completeness-review

BA-canon sweep
Inputs
Every file in input/ via source-manifest.json. Measures against the BA literature canon (IEEE 29148, Volere, BABOK, Wiegers, INCOSE, ISO 25010) across ten coverage dimensions.
When to use
Pick this when you need a broad, authority-grounded gap punch-list to drive client follow-up elicitation. Complementary to gap-analysis.
Delivers
review-inputs/COMPLETENESS-REVIEW/completeness-review.md — disposition-classified findings (Needs-Clarification / Standard-Rule-Applies / Out-of-Scope) plus per-source elicitation questions.
How to use
Use the elicitation questions to chase the client; once you receive answers, add them as new files under input/ and re-run /requirements.

ambiguity-review

Berry/Kamsties + Femmer taxonomy
Inputs
Every file in input/ via source-manifest.json.
When to use
When the input set reads "soft" — vague verbs, weak modals, optional language, undefined pronouns — and you want to nail down each ambiguity before the drafter inherits it.
Delivers
review-inputs/AMBIGUITY-REVIEW/ambiguity-review.md — every lexical / syntactic / referential / vague / subjective / weak-verb / optionality finding with a ready-to-paste stakeholder elicitation question.
How to use
Run the questions past the client; capture answers in input/ (or rewrite the ambiguous source if you have the authority) and re-run /requirements.

gap-analysis

Template-bijection delta
Inputs
Every file in input/ via source-manifest.json. Measures against this project's drafter template — not the BA canon.
When to use
When you want a visual, drafter-aligned gap map with pre-drafted shall-form candidate requirements. Complementary to completeness-review.
Delivers
review-inputs/GAP-ANALYSIS/gap-analysis.html — inline-SVG coverage heatmap, gap-matrix table by dimension × MoSCoW. Every Must/Should gap ships a shall-form Candidate Requirement.
How to use
Copy gap-analysis.html directly into input/ and re-run /requirements — the drafter cites it via its filename and absorbs the candidate requirements. Edit any rows you reject before re-ingestion.
/analyse-inputs Optional · pre-requirements · 11 methodologies
↑ Back to flowchart

Transform raw inputs into structured models (theme maps, task trees, object models, journeys). Every analyser ships a re-ingestion block that survives copy-into-input/ and feeds the drafter on the next /requirements run.

task-analysis

Hierarchical Task Analysis
Inputs
Every file in input/ via source-manifest.json.
When to use
When the inputs describe a goal-decomposition workflow — multi-step tasks with branches, iterations, and information requirements — and you want a numbered tree the drafter can ladder against.
Delivers
analyse-inputs/TASK-ANALYSIS/task-analysis.html — numbered goal / sub-goal / operation tree with mandatory Plans (sequence / selection / iteration / concurrent / discretionary) and per-terminal data-noun annotations. Embedded YAML structured tree.
How to use
Copy the HTML into input/; the drafter's gap-pass treats the embedded YAML tree as a bijection target — Plans seed acceptance-criteria branches, information requirements hint at §7 Data entities.

thematic-analysis

Braun & Clarke six-phase
Inputs
Every file in input/ via source-manifest.json.
When to use
When the inputs are heavy on qualitative material (interview notes, open-ended survey responses, stakeholder narratives) and you want to surface latent patterns before drafting.
Delivers
analyse-inputs/THEMATIC-ANALYSIS/thematic-analysis.md — codes, themes, theme-map (Mermaid), and a Candidate-Requirements bridge linking each theme to draftable scope.
How to use
Copy into input/; the drafter picks the candidate-requirement bridge up like any other source.

opportunity-solution-trees

Teresa Torres 2016 (forward discovery)
Inputs
Every file in input/ via source-manifest.json.
When to use
When the inputs name a desired outcome and you want to ladder it down through opportunities and solutions before the drafter commits to feature scope.
Delivers
analyse-inputs/OPPORTUNITY-SOLUTION-TREES/opportunity-solution-tree.md — four-layer discovery tree (Outcome → Opportunities → Solutions → Assumption-tests) with a Candidate-Requirements bridge.
How to use
Copy into input/; the drafter consumes the bridge section to seed scope. Use the assumption-tests to drive validation work in parallel.

journey-mapping

NN/G + Kalbach current-state journey
Inputs
Every file in input/ via source-manifest.json.
When to use
When the inputs describe an end-to-end user experience (one or more personas) and you need a concrete backbone for §5 task flows, §6 integration constraints, §4 NFRs, user stories, and priorities.
Delivers
analyse-inputs/JOURNEY-MAPPING/journey-mapping.html — one journey card per persona with phases, steps, touchpoints, channels, thoughts, emotion-curve SVG, pain-points, opportunities, moments of truth.
How to use
Copy into input/; HTML survives markitdown round-trip with [SRC: <filename>] citations intact so the drafter inherits the audit trail.

jtbd

Christensen-Moesta + Ulwick
Inputs
Every file in input/ via source-manifest.json.
When to use
When you want the requirements to anchor on user motivation rather than features — "when X, I want Y, so I can Z" — and you have raw inputs that imply jobs without naming them.
Delivers
analyse-inputs/JTBD/jtbd-job-map.html — canonical statements, four forces (push / pull / anxiety / habit), 5×5 Ulwick opportunity matrix.
How to use
Copy into input/; the drafter cites jobs as positive scope and forces as design-influence context.

ooux

Sophia Prater ORCA
Inputs
Every file in input/ via source-manifest.json. Synonym merge across sources is the load-bearing Round 2.
When to use
When the inputs imply a domain object model (entities, relationships, CTAs) and you want it canonicalised — with conflicting names merged and provenance preserved — before the drafter commits to §7 Data.
Delivers
analyse-inputs/OOUX/ooux-object-map.html — Mermaid erDiagram, OOUX sticky-note column-board, relationship matrix, and an embedded JSON object-map body block (load-bearing for drafter re-ingestion).
How to use
Copy into input/; the JSON body and erDiagram both survive markitdown HTML → MD as fenced code so the drafter consumes the full object model in one shot.

swim-lane-process-mapping

Rummler-Brache + Disconnect Analysis
Inputs
Every file in input/ via source-manifest.json.
When to use
When the inputs describe multi-actor processes with handoffs (the "white space" between functional silos) and you want every lane-to-lane handoff classified — clean or one of four disconnect types.
Delivers
analyse-inputs/SWIM-LANE-PROCESS-MAPPING/swim-lane-process-mapping.html — Mermaid flowchart TD per process with subgraph lanes, Actor inventory, Disconnect Register, embedded YAML model.
How to use
Copy into input/; the YAML model becomes a bijection target — processes → §5 task flows, handoffs → §6 integrations, decisions → acceptance-criteria branches, data-stores → §7 entities.

affinity-mapping

Kawakita KJ + Beyer/Holtzblatt
Inputs
Every file in input/ via source-manifest.json.
When to use
When the inputs are a dense pile of atomic claims (interview snippets, sticky-notes, observations) and you want them clustered bottom-up into 4–8 super-themes without imposing a prior frame.
Delivers
analyse-inputs/AFFINITY-MAPPING/affinity-map.html — Mermaid mindmap, cluster cards with stable/drifted confidence chips (Jaccard between two anti-anchoring passes), orphans table, optional tension flowchart, embedded JSON hierarchy.
How to use
Copy into input/; super-theme insight statements feed §1 vision anchors, clusters seed §5 task-flow groupings and acceptance-criteria threads, orphans feed §10 out-of-scope candidates, tensions feed trade-off-aware criteria.

glossary

Ubiquitous-language vocabulary
Inputs
Every file in input/ via source-manifest.json. (Distinct from the same-named /analyse-requirement method, which reads requirements.md.)
When to use
When you want one agreed vocabulary for the system before drafting — significant terms surfaced, classified domain vs application, defined from the inputs, and rated for how settled each shared understanding is.
Delivers
analyse-inputs/GLOSSARY/glossary.html — per-term cards with domain/application + 0–4 maturity badges and cited definitions, plus fenced proposed definitions / canonical resolutions wherever a term is left undefined, weak, synonymous, or conflicting.
How to use
Re-drop into input/ so /requirements adopts the agreed terms as canonical vocabulary; the blocking proposals reach the resolver as questions you confirm before they anchor requirements.

user-goal-analysis

Actor & end-user goals (GORE)
Inputs
Every file in input/ via source-manifest.json.
When to use
When you want the actor and end-user goals behind the request made explicit — both the goals the inputs state and the ones they imply. The actor-goal complement to business-context-definition.
Delivers
analyse-inputs/USER-GOAL-ANALYSIS/user-goal-analysis.html — a goal register grouped by goal type, an AND/OR refinement tree, an actor map, and a conflicts table; inferred goals flagged for confirmation.
How to use
Re-drop into input/ so explicit goals seed §4 user stories and inferred goals reach the resolver as questions you confirm before they become requirements.

business-context-definition

Enterprise motivation (BMM + BABOK)
Inputs
Every file in input/ via source-manifest.json.
When to use
When you need the enterprise motivation behind the request — the business problems, needs, and goals — rather than the individual actor goals user-goal-analysis covers. Run the two side-by-side for two clean, non-overlapping registers.
Delivers
analyse-inputs/BUSINESS-CONTEXT-DEFINITION/business-context-definition.html — a report linking business problem → need → goal → problem-statement in one causal chain; inferred items flagged for confirmation.
How to use
Re-drop into input/ so the causal chain seeds strategic framing and requirement traceability; run user-goal-analysis alongside it for actor-level goals.

/requirements

Citation-grounded system spec
Inputs
Every file in input/, enumerated via the shared source-manifest.json. Multimodal sources (screenshots, diagrams) read natively.
When to use
First proper pipeline of any engagement, once the input drop is complete (and any pre-requirements lensing — PRD, input reviews, input analyses — has been folded back into input/). Re-run after every meaningful change.
Delivers
requirements/requirements.md — citation-grounded, deterministic spec. Inline [SRC: C-NNN] traces stripped on merge; [AI-SUGGESTED] / [STANDARD-RULE] / [OUT-OF-SCOPE] markers retained.
How to use
The downstream contract every other pipeline keys off. Hand to /wireframe as-is. Lens with /analyse-requirement or /review-requirement before sign-off; fold review verdicts back as manual edits.

Brand token set
Inputs

The final, highest-fidelity step. Each run produces one clickable, client-side-only React/Next.js prototype of a scope of requirements.md, accumulating in one shared app under prototypes/ (scaffolded once from template/) reachable from a single landing page. Look & feel is brand-locked and identical across every prototype — divergence is pure UX: a named UX posture + trade-off positions (D1–D5) reshape layout and workflows. Shares the scope-selector skill and blueprint-architect agent with /wireframe.

/prototype

Hi-fi clickable prototype (one shared app)
Inputs
requirements/requirements.md (scoped via the shared scope-selector) + your A–E supporting-input selection — A requirement, B analyse-requirements/*, C analyse-inputs/*, D wireframes/<scope>/*, E input/* — plus design-system/design-system.html for brand (if present; else reference-URL / paste-tokens / template defaults) and template/ as the one-time scaffold source.
When to use
After requirements.md is accepted and wireframes have settled a direction, when you want a clickable artefact to demo with the client. Run it again — different posture or positions, same brand — to add a divergent prototype for side-by-side comparison.
Delivers
prototypes/ — one shared Next.js app: a landing page listing every prototype grouped by scope, a per-run prototype under src/app/<name-slug>/, shared components / fixtures / one theme.css, and a PrototypeChrome with a role switcher (PI-05) + data reset. Every data-bound element is traceable to a blueprint Property closed set (anti-fabrication); the build is verified (lint + typecheck + build + Playwright smoke) and honours invariants PI-01..PI-08.
How to use
Pipeline per run: scope + name → A–E inputs → blueprint (reused if present) → purpose + posture + positions → design-spec draft → resolve → merge → scaffold-if-needed → parallel per-surface generation → verify → landing update. Open the landing page, click into the prototype, and walk the client through the workflow. Designating a /wireframe variant as the basis pre-fills the posture and D1–D5 positions from that variant.
