## Output audience (orientation)

Analysis and review outputs (`/analyse-requirement`, `/analyse-inputs`, `/review-requirement`, `/review-inputs`) are **human-audience** ÔÇö readable without methodology knowledge per `framework/shared/output-readability.md` (an "In plain terms" lead first, first-use jargon glossing, retained citations). Analyses are *additionally* machine-consumed downstream ÔÇö `/analyse-inputs` ÔåÆ `/requirements`; `/analyse-requirement` ÔåÆ `/wireframe` `blueprint-architect` (optional, via the per-analysis sidecar) ÔÇö so their embedded machine-readable sidecar is retained and their machinery prose moves to a collapsed "For downstream use" footer rather than being deleted. Reviews have **no** downstream consumer (machinery text is removed, not footered). `requirements/requirements.md` stays LLM-audience; `prd/prd.md` is human-audience. (Canonical rule: `CLAUDE.md` ┬ğ1 "Output audience".)

## Command mechanics (full)

The slim command catalogue lives in `CLAUDE.md` ┬ğ1. The two heavyweight pipelines are described in full here.

- **`/wireframe`** ÔÇö 2ÔÇô3 parallel low-fi HTML wireframe variants for a scope of `requirements/requirements.md`. Persona-bound; variants diverge on trade-off dimensions **and** information architecture (per-surface realization: standalone screen / inline drawer / inline-expand / modal / wizard-split); divergence axes + persona bindings are **goal-driven** (extrapolated from ┬ğ3 personas + ┬ğ4 goals by `divergence-heuristics.md`, surfaced as a recommended default at the scope-selector confirm gate). Each variant is also **posture-bound**: the divergence heuristic auto-recommends a **UX posture** (`framework/assets/wireframes/design-philosophies.md`, the cross-pipeline registry) per variant, which the architect consumes as a structural/realization + naming overlay (it biases pattern picks but does **not** change the variant's dimension positions, so the comparison stays single-axis). Full requirement-ID traceability via `data-src` + `data-prop`; metadata-only `index.html` landing with side-by-side screen links + a decomposition/structure comparison row. Cross-pipeline `blueprint-architect` agent + `scope-selector` skill + `design-philosophies.md` posture registry reusable by `/prototype`.

- **`/export-application`** (lightweight, listed here because it is new) ÔÇö single-agent, stateless pipeline that re-projects the finished `requirements/requirements.md` to the application audience at `export-application/requirements-application.md`. **Pure transform, zero generated content**: ┬ğ6.10 fixture rows ÔåÆ backend-contract pointer rows, ┬ğ7 `prototype-fixture` ÔåÆ `backend-contract` labels, PI appendix removed, header `Target` flipped, `## Export provenance` block inserted (source sha256 + citation legend). The ┬ğ1.7 / ┬ğ6.6.1 / ┬ğ6.6.2 guidance sections and the ┬ğ6.1 Rationale column pass through verbatim ÔÇö they are drafted in the pipeline doc (see below). Freshness is sha256-anchored: re-invoking compares the embedded source hash against the current `requirements.md` and recommends Keep or Regenerate. Related pipeline change: `/requirements` no longer asks the consultant for a build target ÔÇö `manifest.target` is **auto-set to `"prototype"`** at the orchestrator's Step 1b (legacy `"application"` manifests are honoured by dormant branches), and the pipeline doc now always emits ┬ğ1.7 Architectural implications, ┬ğ6.6.1 Session UX, ┬ğ6.6.2 FE performance budgets (scope-noted application-build guidance ÔÇö never prototype design inputs) plus the per-cell-optional ┬ğ6.1 Rationale column.

- **`/prototype`** ÔÇö One hi-fi, **clickable, client-side-only** React/Next.js prototype per run of a scope of `requirements/requirements.md`, accumulating in **one shared app** (`prototypes/`, scaffolded once from `template/`) reachable from a single landing page. **Look & feel is brand-locked and identical across all prototypes** (one `theme.css` from `/design-system` ÔåÆ consultant ÔåÆ defaults); divergence between prototypes is **pure UX** ÔÇö a selectable named **UX posture** (`framework/assets/wireframes/design-philosophies.md`, the cross-pipeline registry shared with `/wireframe`) + trade-off positions (D1ÔÇôD5) reshape layout + workflows. Reuses `scope-selector` (`propose_divergence_axes:false`) + `blueprint-architect` (`variants_output_path:null`, blueprint-only). Pipeline: scope+name ÔåÆ inputs (A requirement / B analyse-requirement / C analyse-inputs / D wireframes / E input docs) ÔåÆ blueprint ÔåÆ purpose+posture+positions ÔåÆ design-spec **draftÔåÆresolveÔåÆmerge** (*fast path*: when the draft carries zero `[AI-SUGGESTED]` markers ÔÇö common on the posture/wireframe-seeded path ÔÇö the orchestrator skips the resolver and runs the merger in `mode: "mechanical"` with no spec-accept loop; acceptance is the single Step-G prototype gate) ÔåÆ scaffold-if-needed ÔåÆ **parallel per-surface generation** ÔåÆ verify (lint+typecheck+build+Playwright smoke) ÔåÆ landing update. Anti-fabrication via blueprint Property closed sets (`data-prop`); honours PI-01..PI-08.

## Data flow

1. Consultant types `/start` or a specific slash command ÔåÆ command shim ÔåÆ orchestrator adopts persona.
2. Orchestrator runs **preflight gates** (prerequisite check, prior-progress detection, `AskUserQuestion` for start-fresh vs continue / overwrite vs keep).
3. Orchestrator writes `called` event to progress file (where it owns one) + `stage_start` to `timing.ndjson`, then invokes the next agent **in the foreground (same thread)**.
4. Agent reads its inputs per `framework/dependency-graphs.md`, executes its workflow, writes its artefact, calls `verify-artifact-write`, runs self-validation, prompts consultant for accept via `AskUserQuestion`.
5. Agent hands back. Orchestrator writes `completed` + `stage_end`. Repeats per step until DoD.
6. Final acceptance ÔåÆ `status: complete` + `run_end` event.

## Stadium-app input flow (orientation)

A **Stadium 6 application** ÔÇö a deployed Twenty57 Stadium low-code web app, dropped into `input/` either as a folder (`administration.db` + `App_Data/Updates/*.sapz` + `ClientApp/`) or as a one-line `*.stadium` pointer file naming such a folder's path ÔÇö is an optional input *unit* that the standalone `/ingest-stadium` command extracts **once** into a set of lean, citation-ready requirement assets the normal pipelines then consume as ordinary `Native-text` inputs.

- **Ingestion command (`/ingest-stadium`).** The standalone command owns extraction. Its `framework/agents/stadium-ingestor.md` agent detects Stadium units, resolves each app's `app_id` (the app folder basename, equal to its Stadium `FileGuid`), preflights Python (`RF-01`, via `preflight-cli.md` + `setup-instructions/stadium.md`), invokes the extraction skill, and writes the ledger. Its `framework/orchestrators/ingest-stadium-orch.md` surfaces a per-app **re-ingest gate** (re-ingest / skip / cancel) when an app is already ledgered ÔÇö on re-ingest it git-checkpoints, deletes the app's prior assets + ledger entry + forensic model dir, then re-extracts. This command does **not** build the source manifest.
- **Detection / exclusion / nudge (input-handler Step S).** A pre-enumeration step in `framework/agents/input-handler.md` detects Stadium units only to **exclude** the app folder / `*.stadium` pointer from manifest enumeration (step 1) **and** from the freshness check (step 0) ÔÇö they are never manifest rows ÔÇö and to **nudge** the consultant to run `/ingest-stadium` when a detected app is not yet in the ledger. It does **not** extract, preflight Python, or write the ledger. The manifest points only to the extracted assets.
- **Process-once ledger.** `framework/state/.stadium-processed.json` is a runtime ledger keyed by `app_id`, written by `stadium-ingestor`; an already-keyed app is skipped by the ingestor (assets already exist under `input/`, preserving consultant hand-edits) unless the consultant re-ingests it via the orchestrator's gate. A successful extract writes the ledger; a failed extract does not. The input-handler reads it read-only for the nudge.
- **Extraction skill (`framework/skills/extract-stadium-app.md`).** Phase A shells out to the extractor tool for all deterministic sharding; Phase B is a bounded LLM inference that adds two advisory `[AI-SUGGESTED]` assets (`task-flows`, `quality-signals`), verified via `verify-artifact-write.md`.
- **Extractor tool (`framework/tools/extract_stadium_app.py`).** Vendored stdlib-only Python. Its `--emit-assets <dir> --stem <s> --kb framework/assets/stadium --model-out <path>` mode does the sharding in Python and writes ten Tier-1 assets (`<stem>.stadium.{overview,data-model,data-sources,business-rules,access-control,surfaces,navigation,glossary,design-signals,modules}.md`) plus the full forensic `model.json` under `framework/state/stadium/<app-id>/` (never manifested; the LLM never loads it). A sanctioned runtime-code exception, like the markitdown/inkscape ingestion CLIs and `/prototype`.
- **Knowledge base (`framework/assets/stadium/`).** Read-only reference read by the extractor (`--kb`, for module glosses) and conditionally by the drafter: `registry.md`, an app-domain Stadium `glossary.md`, `admindb-schema.md`, `module-catalogue.md`, `theming-model.md`, `template-model.md`, `sapz-spec.md`, `asset-schemas.md`. Its `glossary.md` is **application-domain** Stadium vocabulary, walled off from the system glossary.
- **Where the assets land.** Per-app assets land in `input/<AppName>.stadium-assets/` as ordinary `Native-text` files and are enumerated normally; from there they feed `/requirements`, `/generate-prd`, `/analyse-inputs`, and `/review-inputs` with no further special handling (the drafter conditionally reads `framework/assets/stadium/asset-schemas.md` when Stadium assets are present, and routes advisory/Tier-B lines to `[AI-SUGGESTED]`, never `[SRC]`).

## Per-pipeline writes / reads (full enumeration)

The write-isolation **principle** and the four documented cross-pipeline exceptions are the contract ÔÇö they live in `docs/maintenance.md > Stand-alone constraints (write isolation)`. This is the exhaustive per-pipeline enumeration.

- `/requirements` writes `requirements/*`, `input/*.converted.md`, `framework/state/*` (`.progress.json`, resolver sidecars, `timing.ndjson`). It does **not** write Stadium assets ÔÇö those are produced by `/ingest-stadium` (see below) and consumed here as ordinary inputs; the shared input-handler's Step S only excludes the raw app folder / `*.stadium` pointer and nudges when a dropped app is un-ingested (the same behaviour under `/generate-prd`, `/analyse-inputs`, `/review-inputs`).
- `/ingest-stadium` writes `input/<AppName>.stadium-assets/*.md` (the extracted Stadium category assets + `embedded/`), `framework/state/.stadium-processed.json` (the process-once ledger), and `framework/state/stadium/<app-id>/model.json` (forensic; never manifested). Reads the dropped app folder / `*.stadium` pointer (read-only) and `framework/assets/stadium/*` (the KB). Does **not** touch `requirements/` or build the source manifest. On a re-ingest it also deletes the chosen app's prior assets + ledger entry + model dir (after a git checkpoint).
- `/generate-prd` writes `prd/*`, `framework/state/.prd-progress.json`, `framework/state/prd-resolver-*.{ndjson,json}`, `timing.ndjson`. Reads `requirements/source-manifest.json` + `input/*`. Fully independent of `requirements/requirements.md` ÔÇö never reads or writes it.
- `/design-system` writes `design-system/*`. Reads nothing under `requirements/` or `framework/state/`.
- `/analyse-requirement` writes `analyse-requirements/<METHOD>/*`. Reads `requirements/requirements.md`.
- `/analyse-inputs` writes `analyse-inputs/<METHOD>/*`. Reads `requirements/source-manifest.json` + `input/*` (multimodal for `Native-multimodal` tier). Does not write `framework/state/*`.
- `/review-requirement` writes `review-requirements/<METHOD>/*`. Same read scope as `/analyse-requirement`.
- `/review-inputs` writes `review-inputs/<METHOD>/*`. Same read scope as `/analyse-inputs`. Does not write `framework/state/*`.
- `/wireframe` writes `wireframes/<scope-slug>/*`. Reads `requirements/requirements.md` (scoped by `scope.json`) + `framework/assets/analyses/registry.md` + `framework/assets/analyses/sidecar-schema.md` + selected `analyse-requirements/<METHOD>/*` sidecars (sidecar-first per `framework/assets/analyses/sidecar-schema.md`; bounded prose fallback under `RF-09`; drift halts under `RF-08`) + `framework/assets/{pattern-catalogue,wireframes,design-systems,templates}/**`.
