# Operations Portal Requirements — v1

## 1. Overview

The Operations Portal manages build, deployment, execution, and lifecycle operations for custom applications generated by the platform's AI agents. It is intended for use by technical business users, not professional developers or DevOps specialists. The same users who use AI agents to build frontend and backend applications will use this portal to run and manage what they build.

The portal is designed for consulting companies that deliver custom software to their own clients on a project basis. A consulting company is a tenant, and each client engagement is a project within that tenant. Users are uniquely identified by email address and may be members of one or more tenants. Within a tenant, users are assigned to projects with strict access isolation between projects.

The portal can also be installed on-premise at customers. In this scenario, the customer or consulting company manages the installation and internal business units or teams acts as tenants.

The portal abstracts underlying container orchestration infrastructure, presenting application management in terms that align with business operations rather than infrastructure concepts.

## 2. Key Concepts

| Concept | Definition |
| --- | --- |
| **User** | A person who uses the portal, uniquely identified by email address. A user exists at the system level and may be a member of one or more tenants, with separate roles and project assignments per tenant. |
| **Tenant** | A consulting company or organisation. All resources and projects are scoped to a tenant. Users access a tenant through a tenant membership. The platform supports up to 100 tenants. |
| **Project** | A client engagement or initiative within a tenant. Projects provide strict isolation — users only see projects they are assigned to. Each project contains its own environments, applications, and resources. |
| **Environment** | A configurable logical grouping (e.g., dev, staging, production) within a project. Applications within an environment can communicate with each other. |
| **Application** | A deployable unit of software, consisting of source code in a Git repository. An application is defined at the project level (name, owner, Git repo, metadata) and has zero, one, or more instances across environments. |
| **Instance** | The per-environment running form of an application. Each instance has its own health status, configuration, and resource limits. An application may have one or more instances per environment (e.g., App X in dev, staging, and production are three separate instances). |
| **Build** | The process of compiling source code into a container image. |
| **Container** | A packaged, versioned, runnable instance of an application. |
| **Platform Administrator** | A system-level role responsible for managing the platform itself — creating tenants, assigning initial tenant administrators, and overseeing tenant lifecycle. Platform administrators operate above the tenant level and do not have access to tenant-internal data. |

Note: The term "Service" is reserved for backend business logic services within generated application code (e.g., UserService, PaymentService). In the portal UI, users interact with "Applications" (project-level) and their "Instances" (per-environment).

### Naming Convention

Tenants, projects, environments, and instances must each have an editable display name and an immutable identifier (url_id). The url_id is defined by the user at creation, must be lowercase alphanumeric characters and hyphens, and must be unique within its scope. The url_id is used in all infrastructure references including DNS, URLs, and service discovery. Display names are shown in the UI and may be changed without affecting infrastructure. Applications do not have a url_id — they are referenced by display name and internal id.

## 4. Multi-Tenancy & Project Isolation

| ID | Requirement |
| --- | --- |
| TEN-01 | The portal must support multiple tenants (up to 100) with full data and resource isolation between tenants. |
| TEN-02 | Compute isolation between tenants must be enforced at the infrastructure level. |
| TEN-03 | Each tenant must have isolated databases (no shared database instances between tenants). |
| TEN-04 | Each tenant must have isolated file storage volumes. |
| TEN-05 | Tenant administrators must be able to manage tenant memberships, roles, and projects within their tenant. |
| TEN-06 | Cross-tenant data access must not be possible through the portal or the hosted applications. A user with memberships in multiple tenants must operate in one tenant context at a time, with no data leakage between tenant contexts. |
| TEN-07 | The portal must support tenant-level configuration of tenant name and allowed identity providers. |

### 4.1 Project Isolation

### 4.2 Platform Administration

| ID | Requirement |
| --- | --- |
| PADM-01 | The platform must support a Platform Administrator role that operates at the system level, above all tenants. |
| PADM-02 | Platform administrators must be able to create new tenants by specifying a display name and url_id. |
| PADM-03 | Platform administrators must be able to assign the first tenant administrator when creating a tenant. Subsequent tenant membership management is the responsibility of the tenant administrator. |
| PADM-04 | Platform administrators must be able to view a list of all tenants, including display name, status (active/suspended),creation date, and number of projects. Platform administrators must not be able to view or access tenant-internal data (projects, applications, environments, user data, secrets, or audit logs). |
| PADM-05 | Platform administrators must be able to suspend a tenant. Suspension prevents all tenant members from accessing the tenant and stops all running instances. Suspension does not delete any data. Platform administrators must be able to reactivate a suspended tenant, which restores access and allows instances to be started. |
| PADM-06 | Platform administrators must be able to delete a tenant. Deletion is only permitted if the tenant is suspended and has no running instances, databases, or storage buckets. Deletion requires the administrator to type the tenant url_id to confirm. Deletion is permanent and recorded in the audit trail. |
| PADM-07 | The first platform administrator account must be created during platform installation via a setup configuration or seed process. |
| PADM-08 | Platform administrators must be able to grant and revoke the platform administrator role to/from other users. There must always be at least one active platform administrator — the platform must prevent removal of the last one. |
| PADM-09 | All platform administration actions (tenant creation, suspension, reactivation, deletion, role changes) must be recorded in the audit trail. |
| PADM-10 | Platform administrators must be able to view a per-tenant summary including project count, environment count, running instance count, database count, storage bucket count, membership count, and suspension history. This summary must not expose tenant-internal data (project contents, application data, log content, or full user profile details beyond email and role). |

## 5. Role-Based Access Control (RBAC)

| ID | Requirement |
| --- | --- |
| RBAC-01 | The portal must enforce role-based access control for all operations. |
| RBAC-03 | Roles must be assignable at four levels: platform-wide (platform administrator), tenant-wide (e.g., tenant administrator), per-project (e.g., operator on Project A, viewer on Project B), and per-environment within a project (e.g., deployer in dev but viewer-only in production). |
| RBAC-04 | Permissions must be granular, covering: project management, application management, environment management, data storage management, secrets management, user management, and audit viewing. The detailed action-to-role permission matrix must be defined during the design phase. |
| RBAC-05 | The portal must provide the following default roles: Platform Admin, Tenant Admin, Project Admin, Operator, and Viewer. Custom roles are not supported in v1. |
| RBAC-06 | Role assignments must be audited (who assigned what role to whom, on which project/environment, and when). |
| RBAC-07 | Each tenant must have at least one tenant administrator at all times. The portal must prevent any action that would remove the last tenant administrator — including role reassignment, deactivation, or removal of the last remaining tenant admin's membership. |

## 7. Application Management

| ID | Requirement |
| --- | --- |
| APP-01 | Users must be able to register a new application within a project by linking a Git repository containing source code. The portal must support GitHub and BitBucket as Git providers. The portal must validate that the repository is accessible at registration. If repository credentials later become invalid, builds will fail with a clear error message. |
| APP-02 | Git credentials for accessing repositories must be stored as project-level secrets managed by the portal. |
| APP-03 | Multiple applications may be linked to the same Git repository (monorepo). Each application may specify a subdirectory within the repository. If no subdirectory is specified, the repository root is used. |
| APP-04 | Users must be able to view a list of all applications within their project. |
| APP-05 | Users must be able to view application details including: the application definition (Git repo, metadata), and a per-environment breakdown showing instance build number and health status. |
| APP-06 | Application metadata (name, description) must be editable. |
| APP-07 | Users must be able to delete an application from a project. Deletion is only permitted if the application has no running instances. Deletion requires explicit confirmation and is recorded in the audit trail. |

### 7.1 Resource Linking

| ID | Requirement |
| --- | --- |
| LNK-01 | Users must be able to link a database or object storage bucket to a specific instance. The linked resource and the instance must be in the same environment. |
| LNK-02 | When linking a database, the portal must automatically inject connection details into the instance: non-sensitive values (host, port, database name) as configuration, and credentials (username, password) as secrets. |
| LNK-04 | When linking an object storage bucket, the portal must automatically inject access credentials (endpoint URL, access key, secret key, bucket name) into the instance via configuration or secrets. |
| LNK-05 | Users must be able to view all databases and object storage buckets linked to an instance. |
| LNK-06 | Users must be able to unlink a database or object storage bucket from an instance. Unlinking removes injected connection details and automatically restarts the affected instance. |
| LNK-07 | When a database or object storage bucket is deleted, all links to that resource must be removed. Affected instances are automatically restarted. |
| LNK-08 | Resource link changes (link, unlink) must be recorded in the audit trail. |

## 8. Build & Containerisation

| ID | Requirement |
| --- | --- |
| BLD-01 | The portal must provide a built-in container build system that produces container images from source code without requiring external CI tooling. |
| BLD-02 | Each application must have a configured build branch, set at registration and editable thereafter. Changes to that branch in the Git repository must automatically trigger a build. For monorepo applications, only changes within the application's subdirectory must trigger a build. |
| BLD-03 | Each build must produce a versioned container image tagged with the Git commit SHA. Each build must also be assigned a sequential build number per application (e.g., Build #1, #2, #3). | BLD-04 | Build logs must be captured and viewable in the portal in real time and after completion. |
| BLD-05 | Build status (queued, in progress, succeeded, failed) must be visible to users. |
| BLD-06 | The portal must store build history, including build number, who triggered the build, source commit, build duration, and outcome. |
| BLD-07 | Failed builds must provide clear error output to help users diagnose issues. |
| BLD-12 | All builds must use a standard platform-provided build image with fixed resource allocations and a system-wide timeout. Builds exceeding the timeout must be automatically cancelled and marked as failed. |
| BLD-13 | Users must be able to manually trigger a build for an application. |
| BLD-14 | Users must be able to cancel an in-progress build. Cancelled builds must be marked as cancelled in build history. |

## 11. Instance Management

| ID | Requirement |
| --- | --- |
| INS-01 | Users must be able to deploy a specific build (identified by build number) to an existing instance within their project. This updates the instance's running build via a rolling update. |
| INS-02 | The portal must manage all underlying infrastructure resources on behalf of the user, without exposing orchestration concepts directly. |
| INS-03 | Users must be able to start, stop, and restart an instance. |
| INS-04 | The portal must support rolling updates to minimise downtime when updating an instance to a new version. A new replica must pass its health check before the next old replica is removed. If a new replica fails its health check, the rollout must be automatically halted and rolled back to the previous version. |
| INS-05 | Users must be able to roll back an instance to a previously deployed build. |
| INS-06 | Deployment history must be recorded per instance, including who deployed which build number, when, and the outcome. |
| INS-07 | Users must be able to configure the number of replicas for an instance (manual scaling), from 1 to 10. Scaling to zero is not supported — use the stop action (INS-03) instead, which sets replicas to zero. Starting a stopped instance restores the previously configured replica count. |
| INS-09 | The portal must display instance health status per environment. The defined states are: Running — all replicas are passing health checks; Degraded — at least one replica is passing health checks but one or more replicas are failing; Stopped — the instance has been stopped by a user (zero replicas); Failed — all replicas are failing health checks or unable to start. |
| INS-10 | Resource limits (CPU, memory) must be configurable per instance, within project and tenant-level constraints. Users may select from preset resource profiles (Small, Medium, Large). |
| INS-11 | The portal must provide the following default resource profiles: Small, Medium, and Large. Actual CPU and memory allocations for each profile must be defined during the design phase. |
| INS-13 | The portal must auto-configure a default health check for each instance: HTTP GET on port 8080 at `/health`, expecting a 200 response. Check interval, timeout, and failure thresholds must be defined during the design phase. Users may override the port and path per instance (see INS-14). |
| INS-14 | Users must be able to override the health check path and port per instance. |

## 20. Out of Scope

The following are explicitly out of scope for the Operations Portal:

- **AI agent functionality** — Code generation is handled by separate AI agents, not the portal.
- **Source code editing** — The portal does not provide an IDE or code editor.
- **Billing and metering** — No billing or chargeback. Usage tracking is for internal reporting only (v2).
- **Environment promotion workflows** — Promotion between environments is the customer's responsibility; the portal supports deploying specific versions to specific environments.
- **Infrastructure provisioning** — The portal assumes a running container orchestration cluster is provided; it does not provision clusters.
- **Database schema migrations** — Handled by applications at startup, not the portal.
- **CLI tooling** — Not in scope for initial release (may be considered in future).
- **Advanced deployment strategies** — Blue-green, canary, and traffic-splitting deployments. Rolling updates (INS-04) are the only supported strategy.
- **Cross-region failover** — Multi-region redundancy and automatic failover.
- **Internationalization/localization** — The portal is English only.
- **File volumes** — Persistent file system volumes are deferred to v2. Object storage is the v1 storage option.
- **Auth proxy** — Authentication proxy for hosted applications is deferred to v2.
- **Alerting and webhooks** — Deferred to v2.
- **Custom domains and advanced API gateway routing** — Deferred to v2.
- **Usage tracking and reporting** — Deferred to v2.
- **REST API** — Public REST API for automation is deferred to v2.
