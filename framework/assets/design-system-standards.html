Design System Standards

Universal rules for productivity / data-management applications generated by this framework. Tokens above describe the brand's surface; the standards below describe how components, motion, content, and interactions behave on top of those tokens. These rules are static — identical in every generated design-system doc.

How to read this section. The Component Library defines minimum features per component, not a required roster. A prototype uses whichever components its scope demands; whatever it uses must meet the minimums here.

1. Component Library (Minimum-Feature Contracts)

The visible part everyone expects — but it's only one layer.

This is a baseline contract, not an inventory. A prototype may use a larger or smaller set of components depending on its scope. Whatever it uses must satisfy the minimums below. Components not listed here still inherit the universal rules in sections 2–5 (animation, interaction, accessibility, content).

For each component:

  • Purpose (when to use / not to use)
  • Variants (states, sizes, types)
  • Behaviour (interaction rules and minimum features)
  • Accessibility requirements
  • Do's and don'ts

If you only show visuals and skip behaviour, devs will guess.

Tables

  • Purpose. Display rows of comparable records for scanning, sorting, and acting upon. The default container for data-management views. Do not use for marketing layouts, dashboards-with-tiles, or visual records where the image carries the meaning (use a card grid).
  • Variants. Compact / comfortable density; striped / unstriped; with / without selection; with / without inline filters; with / without grouped headers.
  • Behaviour — minimum features:
    • Every column is sortable. Sort state is visible on the active column header (asc / desc / none) and persists across paginations.
    • The footer always contains pagination controls.
    • Pagination controls include explicit Next and Back navigation.
    • Pagination displays the current page number (e.g. "Page 3 of 12").
    • Pagination includes a page-size selector with options 5 / 10 / 20 / 50, default 20.
    • Empty, loading, and error states are explicit — never an absent or silent table.
    • Long text truncates with tooltip-on-hover; row height stays uniform across the page.
    • Rows are NOT clickable. Row-based actions appear in a dedicated action column as explicit buttons or links — never as a whole-row click target. (Whole-row click is an invisible affordance, ambiguous against text selection, hostile to keyboard users, and conflicts with checkbox / multi-select interactions.)
    • Numeric columns are right-aligned; text columns are left-aligned.
  • Accessibility. Column headers are real <th>. Sort state is announced to assistive tech (aria-sort). Keyboard users can move between cells, trigger sort, change page, and activate row actions without a mouse. Pagination controls are reachable and labelled.
  • Do's and don'ts.
    • Do show pagination controls even when there is only one page (consistency over cleverness).
    • Do let users know how many records exist in total.

Date fields

  • Purpose. Capture a date (or date-time) value in any form.
  • Variants. Date only / date + time / date range; inline calendar / popover calendar / dropdown selector.
  • Behaviour — minimum features:
    • All date form fields are date selectors (calendar pickers). Free-text date entry on its own is not acceptable as the only input mode.
    • The selector accepts keyboard entry as an accelerator (typed date is parsed and reflected in the calendar), but the visible control is the picker.
    • Format is locale-aware; the placeholder shows the expected format ("DD/MM/YYYY" or per locale).
    • Min / max constraints (where applicable) disable out-of-range dates rather than silently rejecting on submit.
  • Accessibility. The picker is fully keyboard navigable (arrow keys move days, PgUp/PgDn move months). The currently focused date is announced. The associated label is visible.
  • Do's and don'ts.
    • Do prefill sensible defaults (today, last selection, business-rule default).
    • Don't ship a free-text date input as the primary control.

Buttons

  • Purpose. Trigger an action.
  • Variants. Primary / secondary / tertiary / destructive; sizes (sm / md / lg); with / without leading icon; loading state.
  • Behaviour — minimum features:
    • Label is verb-led and action-specific ("Save changes", not "OK").
    • Disabled and loading states are visually distinct from each other and from default.
    • Activatable via mouse, touch, Enter, and Space.
    • Destructive variants are visually distinct (colour + icon, never colour alone).
  • Accessibility. Real <button> elements (or <a> for navigation). Visible focus ring. Disabled state communicated to assistive tech.
  • Do's and don'ts.
    • Do limit primary buttons to one per surface.
    • Don't use a button when the action is navigation — use a link.

Inputs / form controls

  • Purpose. Capture user input on a form.
  • Variants. Text / number / email / password / search / textarea / select / checkbox / radio / toggle.
  • Behaviour — minimum features:
    • Every field has a visible label. Placeholders are NOT a substitute for a label — they may complement it but never replace it.
    • Every field has a suitable placeholder that hints at the expected format or example value (e.g. "name@company.com", "DD/MM/YYYY", "Search by ID or name…").
    • The active (focused) field is visually identifiable — clearly distinct from the resting state, using more than colour alone (border weight / focus ring / token-driven outline).
    • Required fields are indicated visibly and programmatically.
    • Helper text and error text have dedicated slots; error state is communicated by colour, icon, and message text (never colour alone).
    • Validation runs on blur for inline feedback; on submit for the form as a whole. Don't validate aggressively on every keystroke.
  • Accessibility. Labels are programmatically associated (<label for> or wrapping). Errors are announced via aria-describedby / aria-invalid. Inputs are reachable via Tab in a sensible order.
  • Do's and don'ts.
    • Do let users paste, autofill, and use password managers freely.
    • Don't disable copy / paste on any field.
    • Don't rely on placeholder text as the only label.

Navigation elements

  • Purpose. Move the user between top-level sections, sub-sections, and views; show where they are.
  • Variants. Sidebar / top nav / breadcrumbs / tabs / segmented control.
  • Behaviour — minimum features:
    • The current location is clearly indicated.
    • Fully keyboard navigable; Tab order matches visual order.
    • Supports deep-linking — the URL reflects the current view so it can be shared, bookmarked, and reloaded without losing state.
    • Navigation never traps users in a sub-flow; there is always a way back.
  • Accessibility. Landmarks (<nav>) are used. Active item is communicated to assistive tech (aria-current).
  • Do's and don'ts.
    • Do persist navigation state (collapsed sidebar, selected tab) across reloads where it improves productivity.
    • Don't hide primary navigation behind a hamburger menu on desktop.

Feedback components (alerts, toasts, banners, modals)

  • Purpose. Communicate system state, action outcomes, errors, and confirmations.
  • Variants. Toast (ephemeral) / banner (persistent) / inline alert (in-context) / modal (blocking).
  • Behaviour — minimum features:
    • Severity is communicated by colour AND icon (never colour alone).
    • Toasts are dismissible; only non-critical info auto-dismisses; errors and warnings persist until acknowledged.
    • Banners persist until the underlying state clears.
    • Modals trap focus while open and close on Esc. Background is non-interactive while a modal is open.
    • Only critical, blocking interactions use modals — never use a modal for non-blocking information.
    • Elevation & edge. A modal must be visually separable from the content behind it: it carries both an elevation shadow and a 1px border, so its edge stays distinguishable even when the modal surface and the page background are the same colour.
  • Accessibility. Live regions (role="status" / role="alert") announce dynamic feedback. Modals expose role="dialog" with a labelled title. Focus returns to the triggering element on close.
  • Do's and don'ts.
    • Do tell the user what happened, why, and what they can do next.
    • Don't stack multiple modals.
    • Don't use a toast for a critical error — use a banner or modal.

2. Animation Policy

  • Subtle by default. Animation is a usability tool, not decoration.
  • Add animation only when it measurably improves usability — to clarify a state change, communicate cause-and-effect, or smooth a transition that would otherwise feel jarring. If an animation has no clear usability justification, omit it.
  • Productivity-app bias. Prefer instant feedback over animated feedback. Avoid animations that delay task completion or sit between the user and their work.
  • Duration tiers map to the brand's transition-fast / transition-base / transition-slow tokens. Default to transition-fast unless there is a reason to slow down.
  • Easing defaults to the brand's easing-standard token.
  • prefers-reduced-motion. Honour the user agent setting. Disable all non-essential motion; keep only state-feedback indicators (e.g. focus ring transitions, spinner for in-flight requests).
  • No parallax, decorative scroll-driven motion, or attention-grabbing entrance animations on data-management surfaces.

3. Interaction Behaviour Standards

  • Required states for every interactive element: default, hover, focus-visible, active, disabled, loading. All six must be designed and implemented.
  • Focus visibility is non-negotiable. Never remove the focus ring without an equivalent indicator. The focused element is always identifiable without a mouse.
  • Loading vs. disabled are visually distinct states with different meanings — disabled means "not allowed right now", loading means "working on it".
  • Keyboard parity. Every mouse-driven action has a keyboard equivalent. Productivity apps are operated by keyboard-heavy users; Tab order is predictable, Enter activates default actions, Esc cancels / closes.
  • Destructive actions require confirmation. Confirmations name what is being changed ("Delete invoice INV-1042?"), never a generic "Are you sure?".
  • Optimistic UI is acceptable for low-risk actions; high-risk actions wait for server confirmation before reflecting state.

4. Accessibility Standards

  • WCAG 2.1 AA is the baseline. Text contrast is already enforced upstream by token validation; the rules below cover the rest.
  • Visible focus indicators on every interactive element.
  • Minimum 44 × 44 px touch targets on touch surfaces.
  • All form inputs have programmatically associated labels (no placeholder-as-label).
  • ARIA roles only when no native semantic exists. Prefer <button>, <a>, <nav>, <table> over <div role="...">.
  • Live regions for async feedback (toasts, validation results, async loads).
  • Tables: column headers are real <th>; sort state is announced (aria-sort).
  • Forms: errors are linked to their field via aria-describedby and the field is marked aria-invalid when in error state.
  • Content order in the DOM matches visual order — assistive tech traverses the page the same way a sighted user does.

5. Content & Microcopy Guidelines

  • Voice & tone. Clear, calm, second-person. Productivity-app users are working, not browsing — say what the user can do, then get out of the way.
  • Button labels. Verb-led, sentence case, action-specific. "Save changes" beats "OK". "Delete invoice" beats "Confirm".
  • Error messages. State what happened and what to do. Never blame the user. ("We couldn't save your changes — check your connection and try again.")
  • Capitalisation. Sentence case for UI labels, headings, button text, navigation items. Reserve title case for proper nouns.
  • Dates. Locale-appropriate format. Use relative for recent events ("2 minutes ago", "Yesterday"); absolute for older. Show absolute on hover when relative is the primary display.
  • Numbers. Thousands separators. Currency with explicit symbol. Right-align numeric columns in tables.
  • Empty states. Tell the user why it's empty and what to do next. Never ship a blank table or page.
  • Loading copy. Brief and honest ("Loading…" not "Please wait while we prepare your experience").

6. Pattern Usage Rules

  • Modal vs. drawer. Modal for blocking confirmations and irreversible decisions. Drawer for context-preserving detail or secondary editing where the underlying view is still useful.
  • Table vs. card grid. Table for comparison and scanning across many records — the default for data-management. Card grid only when the visual record itself carries information (gallery, image library, person directory).
  • Toast vs. banner vs. inline alert. Toast for ephemeral confirmation of an action the user just took. Banner for persistent system state ("System maintenance at 9pm UTC"). Inline alert for in-context validation or warnings tied to a specific element.
  • Filters vs. search. Filters when the dimensions are known and finite (status, owner, date range). Search when the user knows a value (name, ID, free text).
  • See framework/assets/pattern-catalogue/ for canonical pattern definitions referenced above.

7. Application Notes

  • These standards are static and domain-agnostic within the productivity / data-management application class. They do not vary by brand, domain, or generated tokens.
  • When a specific prototype needs an exception, document it in the design-spec (framework/assets/template-design-spec.md), not by editing the design-system output.
  • Components not covered above still inherit the universal rules in sections 2–5 (animation, interaction, accessibility, content).
