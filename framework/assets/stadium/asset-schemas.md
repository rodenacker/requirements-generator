# stadium/asset-schemas.md — per-app extracted-asset schemas & drafter mapping

The canonical schema-doc for the **per-app** assets the Stadium extractor emits, plus the thin contract the requirements drafter follows when citing them. (For the platform concepts behind these assets see `glossary.md`; for the per-app assets' *origin* see `sapz-spec.md` + `admindb-schema.md`.)

When the extractor runs with `--emit-assets <dir> --stem <Stem>`, it writes one lean markdown file per category as **`<Stem>.stadium.<category>.md>`** into that engagement's `input/` directory. Two tiers of asset exist:

- **Tier-1 (deterministic, Python)** — written by `emit_assets()` in `extract_stadium_app.py`. Eleven files: `overview`, `data-model`, `data-sources`, `business-rules`, `access-control`, `surfaces`, `tasks`, `navigation`, `glossary`, `design-signals`, `modules`.
- **Tier-2 (LLM-inferred, advisory)** — added on top by the extraction skill: `task-flows`, `quality-signals`. These are inference, not extraction; they are advisory throughout.

## (a) The YAML provenance header

Every Tier-1 asset opens with the same frontmatter (`_prov_header()`):

```yaml
---
stadium_asset: <category>          # the category slug, e.g. data-model
app: <App name>                    # from design model / admin.db / appsettings
file_guid: <FileGuid>              # app's stable identity = folder name
designer_version: <x.y.z.b>        # Stadium Designer build that published it
selected_package: <GUID>.sapz      # which deployment was read
deployment_count: <N>              # number of .sapz deployment packages found (forensic; §4.1)
last_published: <ISO datetime>     # datetime of the latest deployment (forensic; §4.1)
extracted_from: <abs app folder>   # provenance trail
provenance: deterministic extraction from the Stadium 6 design model + administration.db
marker_legend: Tier-A lines are authoritative facts ([SRC]-quotable); Tier-B lines are advisory design signals.
---
```

The header is forensic provenance: it ties every asset back to a specific app + deployment so a fact can be audited. (Tier-2 assets carry the same header shape, written by the skill.) **The `designer_version` / `selected_package` / `deployment_count` / `last_published` fields are the canonical home of the deployment forensics** — the 2026 redesign (§4.1) removed the duplicate forensic lines from the `overview` *body*, keeping them here only (zero information loss; the body is leaner).

## (b) Tier-A vs Tier-B + the inline locators

Inside each asset, content is split by **explicit `## Tier-A —` / `## Tier-B —` headings**:

- **Tier-A** = provably present in source. Authoritative. Quotable as `[SRC: <asset-filename>]`.
- **Tier-B** = advisory design signal (layout, control-choice, inferred domain). Never authoritative.

Tier-A lines carry an **inline locator** in square brackets naming the exact source within the app:
- `[from design model]` / `[from connector: <FnName>]` / `[from stored-procedure: <ProcName>]` / `[from web-service: <METHOD> <path>]` / `[from script, owner: <Page>]` / `[from administration.db]` / `[from admin.db: <Table>]` / `[from appsettings]` / `[from design model: Setting]` (the `.sapz` `Setting` config table — Cluster B #4-B) / `[from global-scripts]` (module-driven behaviours — Cluster B #5).
- **Rendered `ClientApp/src` locators (Cluster C — see `rendered-clientapp-spec.md`):** `[from rendered routes]` (`page-routes.js` — titled page list + route reachability), `[from rendered types]` (`types.js` — FE↔API §7 fields + per-field authority + nested relations), `[from rendered view]` (`views/*.vue` — deployed-vs-design column divergences + page→connector-function endpoints). Rendered facts are a **complement** to the `.sapz` (reconciled against it; the `.sapz` wins on type conflict) and are distinct from `[from design model]` so an audit can see a fact came from the generated artifact.
- The three data-model locators (`connector` = SQL query/view, `stored-procedure`, `web-service`) plus `rendered types` name which **evidence class** grounds a field, so a reviewer can weight it (SQL columns and proc/endpoint field-names are all Tier-A; web-service field *types* are often weak — see §(e)). They are byte-stable, sanitized renderings and must be quoted **verbatim** (grounding is byte-exact).

Tier-B / gap / inferred lines carry a provenance **marker** instead: `` `[AI-SUGGESTED: domain inference]` ``, `` `[AI-SUGGESTED: blocking]` `` (for gaps), `` `[AI-SUGGESTED]` `` (for implications). These mirror the system's own marker vocabulary (`CLAUDE.md` §2).

### Tier assignment for the behaviour / actor-model enrichment (load-bearing)

The behaviour-enrichment sub-sections (0a–0h, 1a–1e) split fact from interpretation on the **groundedness** axis. What is a fact vs a reading:

- **Tier-A (fact, no marker):** the affordance's control + verbatim **label** + wired script (0e); the **validator** target + rule + verbatim message (0f); the **trigger token** in the script name (`.Click`/`.Load`) and its user-initiated/automatic classification (0g); the ordered actions + **branch tree** structure (1a/1c); the guard **predicate** operands (1b); the notification **message text + `NotificationType`-decoded `success/error/info`** (0a); the **state toggle / error-type / guard predicate** facts (1d); page name + `IsStartPage` + reachability (0b/0c); RBAC roles, grants, counts, matrix, and the single-operator **posture statement** itself (0d); the journey **edges/gestures/page-kinds** (1e); the scaffold **skeleton** (task-clusters + surfaces + notifications per actor, 0h).
- **Tier-B (`[AI-SUGGESTED]`, never `[SRC]`-citable):** the candidate **task** named from a verb (0e); the **goal** a gesture serves (0g); the inferred page **kind** from the suffix taxonomy (0b); the operator **symbol** where the enum was best-guessed (1b — `op<N>` when unmapped); the state **classification** `{loading/empty/error/permission}` (1d); the **capability split** and **actor-candidate** reading — task-cluster→distinct-operator, recipient→second-actor (0d); the journey **narrative** (that a chain *is one* end-to-end task, 1e).
- **The actor / persona scaffold's two-tier shape (0h):** its **skeleton is Tier-A** (the clusters/surfaces/notifications are joined from source facts); its **narrative fields — `name`, `goal`, `motivation`, `pain-points`, `success-metric` — are rendered as empty `` `[AI-SUGGESTED: blocking]` `` gap prompts, never as asserted content.** Authoring those at ingest would fabricate — the `.sapz` cannot ground them. They become resolver questions downstream; persona *narrative synthesis* stays a `/analyse-*` and `/requirements` concern. PII guard: the recipient signal surfaces only *that* an external recipient exists (+ a role-ish label), never an address; actors are named neutrally (`Operator A`, `Role: <name>`).

## (c) Citation discipline for the drafter (load-bearing)

This is the contract the `/requirements` drafter must honour when ingesting these assets:

1. **Tier-A lines are authoritative and `[SRC]`-citable.** Cite them as `[SRC: <Stem>.stadium.<category>.md]` (the manifest-row `filename` payload, per the `[SRC: <filename>]` convention in `CLAUDE.md` §2). **Quote verbatim, including the inline `[from …]` locator**, so the downstream consumer can trace the fact to the exact connector/page/table.
2. **Tier-B, advisory, and `[AI-SUGGESTED]` lines must NEVER be cited as `[SRC]`.** They are mined as *requirement signals* (the *how*, per `CLAUDE.md` §1) — they bias design, they do not bind it. A `how`-decision that honours one cites the *signal*; it does not promote it to an authoritative fact. Gap lines (`[AI-SUGGESTED: blocking]`) become resolver questions, not requirements.
3. **The closed property set comes from Tier-A `data-model`.** Object properties the wireframe pipeline depends on (the `Shape.Field` closed set, `CLAUDE.md` §1) are exactly the Tier-A `data-model` **fields** — never widen them from a Tier-B inference. API/proc-derived fields enter the closed set **by name** (a field's *type* is omitted when it did not resolve concretely — never emitted as `Object`); web-service **endpoints** (method/URL/host) are handoff-only `data-sources` material, **not** §7 properties. Likewise the Cluster A **control enums (`Options`) and DataGrid column labels** in `surfaces` are *surface* facts the drafter maps onto an **existing** §7 data-model property's `ui_display`/`Enums` — they **never mint a new §7 property** (a column carries no `DataField`, and an inline enum is a control setting, not a persisted field). **Cluster C exception (`[from rendered types]` only):** rendered `types.js` fields **do** enter the closed set as Tier-A `data-model` properties — this is deliberate (it fixes web-service apps the `.sapz` left near-untyped), reconciled (not blindly added), provenance-stamped, and the `.sapz` type wins on conflict. Cluster C's other two locators add no §7 property: `[from rendered routes]` (pages/titles/routes) and `[from rendered view]` (column labels/visibility/endpoints) are surface/nav facts, exactly like the Cluster A surface facts above.

## (e) Data-model reconciliation & safety rails

The `data-model` asset is built by `reconcile_entities()` in `extract_stadium_app.py`, which unions entities across **four evidence classes** — SQL connector queries/views (`_sql_shape`), stored procedures (`_sp_shape`: entity+op from the `<prefix>_[Stadium_]<Entity>_<Op>` name, fields from parameters), web-service calls (`_ws_shape`: entity name from the Body/Response `GuidReference → CustomType.Name`, fields from the function's bound Parameter/DataResult struct), and — **Cluster C #6** — the rendered `types.js` FE↔API contract (`read_client_types`; see `rendered-clientapp-spec.md`). The design intent is the **most complete field set per entity**. The rails that keep this union truthful:

- **Union by normalized name (aggressive).** One entity per normalized name; SQL `Members` + API `member` collapse, but distinct DTO variants (`member` vs `fullmember`) keep separate stems and are cross-listed as advisory `related shapes`, never merged. A field may therefore be broader than any single backend record — that is intentional completeness, not fabrication, because…
- **…every field is real and carries per-source `[from …]` locators.** Nothing is invented; the drafter cites each field to its exact SQL/proc/endpoint source. Multiple sources for one field are recorded (the asset shows the first locator + a `(+N more)` note).
- **Envelope/transport filtering.** `_ENVELOPE_DENY` (`success, message, raiseexceptions, statuscode, responsebody, apiresponse`) drops status/transport wrappers from proc params + web-service payloads; a broader `_ENTITY_DENY` (adds `result, error, response, data` **and the rendered envelopes `defaultresponse, generalresponse, generalerror, generalresponseid, responseid, responsemessage, validationresult`**) blocks transport/scratch entity names from **all** sources — this also removes SQL `DECLARE @result TABLE(…)` scratch table-variables and keeps the rendered envelopes out of §7 regardless of which source names them (DA#4: 0 envelope entities across all 20 corpus apps). The field denylist is deliberately narrow so real columns (e.g. a multi-tenant `ApplicationID` FK) are **kept**.
- **Rendered `types.js` evidence + variant consolidation (Cluster C #6).** Rendered fields carry **no type** (all `= undefined`), so they never override a `.sapz`/SQL type — they only ADD to the closed set (type precedence unchanged; `rendered-types` ranks below `sql`). The class `<Entity><Variant>` name is variant-stripped to a clean stem (`UserRead`/`UserWrite` → `User`; `Basic` binds to Read; `Status` stays; `*List`/`*Validation`/envelopes excluded — `rendered-clientapp-spec.md`). A consolidation pass (`_variant_norm`, **gated on rendered types being present** so pure-SQL apps stay byte-identical) then folds the pre-existing web-service shapes keyed on raw type names (`CustomerRead`, `CustomerWrite`, `BeneficiaryReadList`) onto that stem — de-duplicating the rendered↔WS split and cleaning the sparse-WS-entity sprawl. Each rendered field also carries a per-field **authority** (display/editable/action, Tier-B) read from the variant.
- **Weak types omitted.** A field emits `: Type` only when a concrete design-model / SQL / proc type resolved; `System.Object` (common for API shapes) and unresolved types are dropped, leaving the field **name-only**. Type precedence: design-model `SimpleDataType` > SQL/proc `DbType` > web-service > omit.
- **`IsDataField` is NOT a gate.** The flag is uniformly `0` across deployed `.sapz` models in practice, so gating on it would drop every field; the envelope denylist is the real guard (see `sapz-spec.md`).
- **Unclassified stored procedures** (name yields no recognized entity/op) are listed under a Tier-B heading, never fabricated into an entity.

## (f) The 2026 extraction redesign — de-noise + new extractors (reference)

The redesign (`plans/stadium-extraction-redesign/`) raised **grounded claims per token** without changing the asset taxonomy (the 11 Tier-1 + 2 Tier-2 names are unchanged; the `modernization-ux` taxonomy re-wire is a deferred Phase-3 concern). Two structurally distinct moves, kept un-conflated:

- **De-noise (§4.1 — token-cost win).** `surfaces` drops the pure-layout control dump (~−61% corpus-wide) keeping only meaningful controls + a per-page "Visible terms" line + a one-line depth summary; `business-rules` folds UI-plumbing `SetValue`s out of the flow into a compact 1d state counter; `overview` drops deploy-forensic + orientation lines (retained in the YAML header). Removes ~zero authoritative (`[from …]`, non-`[AI-SUGGESTED]`) fact lines.
- **New extractors (§4.3/§4.4 — quality win, grows the numerator).** Entity **relationships** in `data-model` §2.2; a **permission model** in `access-control`; **reports & dashboards** in `surfaces`; **connector kind / maturity / tech-stack** in `data-sources`; **modernization & UX signals** in `modules`; **classified inline JS + a bespoke-rules catalog** in `business-rules`.

**Corpus-invariance omit list (§1.4 — never an extraction target).** Byte-identical across all 20 corpus apps ⇒ Stadium framework shell, never mined: `Controllers/`, `Models/`, `Services/`, `ClientApp/src/{stores,utils,components,setup,assets}/`, `views/{administration,authentication,layout}/`, `.csproj`, `Program.cs`, `Startup.cs`, `web.config`, `package.json`, `App.vue`, `main.js`, `router/{index,routes}.js`. App-specific (`distinct`>1) ⇒ signal (mined): the `.sapz`, `administration.db`, `appsettings.json`, `types/types.js`, `router/page-routes.js`, `global-scripts.js`, `views/*.vue` top-level.

**Determinism / safety rails unchanged.** Phase-A stays pure Python; every new line carries a `[from …]` locator (Tier-A) or an `[AI-SUGGESTED]` marker (Tier-B — e.g. FK-by-naming-convention relations, modernization signals, state classifications); PII + input-safety invariants are untouched.

Correctness of this reconciliation (entity separation, no envelope fields, weak-type omission) is the **extractor's** responsibility and is **not** caught by the skill's Phase-A parse-check (which only asserts each asset exists and is non-empty).
