# map-five-whys-to-ui.md

**Purpose:** Translate a Five Whys justification analysis (`analyses/FIVE-WHYS/five-whys.md`) into signal-based inputs for the design and requirements-quality pipelines. Unlike `map-task-flows-to-ui.md` (which maps decomposition → affordances) or `map-ooux-to-ui.md` (which maps objects → screens), Five Whys produces **signals about requirement-rationale health**, not direct UI affordances — analyses are extraction; design and consolidation are downstream concerns. The mapping is therefore: **thin-justification chains** (high `ai-suggested` density or INCOMPLETE termination) → flagged for consultant interview before the design phase consumes the requirement (a clause without traceable justification is unsafe to design against; the interview surfaces the missing driver); **coverage gaps** (root justifications named in a chain but not anchored in `§1` or `§6`) → flagged for `/requirements` enrichment (the consultant adds the cite to `§1 Application context`, re-runs `/requirements` to refresh the merged doc, then re-runs `/analyse five-whys` to verify the gap is closed); **cross-requirement consolidation candidates** (multiple requirements sharing a root justification per Round 7) → flagged for `/requirements` consolidation review (the consultant either consolidates the requirements in `§6` or documents the shared driver explicitly in `§1`); **policy-driven requirement chains** terminating at regulatory cites (`GDPR`, `PCI-DSS`, `SOX`, `HIPAA`, `ISO`, `NIST`, `FCA`, `MiFID`, internal four-eyes / audit policy) → routed to compliance-pattern-catalogue entries in the design system (audit-log surfaces, retention-banner patterns, two-person-approval modals); **business-goal requirement chains** terminating at quantified targets (KPIs, percentages, throughput numbers) → routed to KPI-dashboard-pattern entries and to outcome-instrumentation patterns; **operational-capability requirement chains** terminating at integration / volume drivers → routed to integration-status patterns (third-party-service health surfaces, throughput indicators); **workflow-constraint requirement chains** terminating at risk / approval / domain rules → routed to gated-CTA patterns (conditional sections, prerequisite checks, approval-required affordances).

**Inputs:** `analyses/FIVE-WHYS/five-whys.md`, `framework/assets/taxonomy-signals.md` (if present), `framework/assets/pattern-catalogue/` (for compliance / KPI / integration / gated-CTA pattern entries).

**Outputs:** Requirements-quality signal entries (thin-chain flags, coverage gaps, consolidation candidates) for the requirements-quality consumer; pattern-catalogue trigger entries for the design-spec consumer.

**Used by:** `framework/agents/design-spec-drafter/agent.md` (when authored) and any future requirements-quality reviewer that consumes analyser output.

**Used how:** Called when the five-whys analysis is present. Combines with other `map-*-to-ui` skills via the accrue-all pattern — each skill contributes complementary entries to the same downstream inventory. The five-whys skill focuses on **justification-health signals and policy/KPI pattern routing**; the task-flows skill focuses on **single-actor screen sequences**; the activity-diagram skill focuses on **multi-actor swimlane partitioning**. Five Whys never authors UI; it only surfaces signals that other consumers act on.
