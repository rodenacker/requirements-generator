# map-swim-lane-process-mapping-from-inputs-to-ui.md

**Purpose:** Translate a Rummler-Brache cross-functional process map + Disconnect Analysis of raw inputs (`analyse-inputs/SWIM-LANE-PROCESS-MAPPING/swim-lane-process-mapping.html`) into signal-based inputs for the downstream requirements-drafting and design-spec pipelines. Unlike `map-activity-diagram-to-ui.md` (which maps the analyse-requirements-side UML 2.5 activity-diagram artefact into screen-flow inventory entries) or `map-task-analysis-from-inputs-to-ui.md` (which maps single-actor hierarchical task decompositions), swim-lane process mapping of raw inputs produces **structural signals about cross-functional workflows, actor ownership, lane-crossing handoffs, and the gaps in those handoffs (the Disconnect Register)**, not direct UI affordances — analyses are extraction; design and consolidation are downstream concerns. The mapping is therefore: **YAML structured model** (the `<pre><code class="language-yaml">` block inside `analyse-inputs/SWIM-LANE-PROCESS-MAPPING/swim-lane-process-mapping.html`, carrying `actors / processes / steps / handoffs / disconnects` with `[SRC: <filename>]` markers throughout) → routed primarily to the **consultant** (via the visible Disconnect Register as the next-elicitation surface — every `consultant_follow_up: yes` row carries a `suggested_question` the consultant brings back to the source author) AND secondarily to the `/requirements` drafter via re-ingestion through `input/` (the consultant copies the HTML into `input/`; input-handler classifies as `Supported-via-MCP`; markitdown converts to `.converted.md` preserving the `<pre><code>` block as fenced code and the `<pre class="mermaid-source">` Mermaid blocks as plain `<pre>` text; drafter reads the YAML structured model as a bijection target for `framework/skills/completeness-gap-pass.md` — every process should map to ≥ 1 `§5 Task flows` entry; every handoff should map to ≥ 1 integration / acceptance criterion in `§6 Requirements`; every disconnect should be either closed by the consultant before drafting or surfaced as a resolver question); **Disconnect Register rows with `consultant_follow_up: yes`** → routed to resolver questions in the `/requirements` resolver pipeline (each carries an `AI-NNN`-eligible suggested question that flows into the existing resolver grammar without schema changes); **Disconnect Register rows with category `unstated-exception`** → routed specifically to `§6 Requirements` acceptance-criteria branches (every unstated exception is a missing *"Given X, When Y fails, Then Z"* row the drafter must elicit); **Disconnect Register rows with category `conflicting-source`** → routed as **blocking resolver questions** because two consultant sources materially disagree (cannot be silently merged); **Disconnect Register rows with category `missing-actor`** → routed to `§3 Target users` and `§4 User goals & stories` as persona / role candidates needing naming before downstream design can allocate ownership; **Disconnect Register rows with category `ambiguous-trigger`** → routed to `§6 Requirements` integration constraints and `§5 Task flows` as integration-contract candidates needing trigger-event clarification; **actor inventory** (every entry in `actors` with `kind: role` / `system` / `external-service`) → routed to `§3 Target users` (role-kind actors → persona seeds), `§2 Domain model` aggregate-root candidates (system-kind actors → internal services), and `§2 Domain model` external-system aggregate-root candidates (external-service-kind actors → integration points); **process step types** → routed by type: `data-store` steps to `§7 Data entities` candidates (every data-store noun is a precursor to a data-model node), `external-system` steps to `§2 Domain model` external-system aggregate-root candidates (every external service is a precursor to an integration constraint in `§6`), `decision` steps to `§6 Requirements` acceptance-criteria branches (every decision's branch guards become *"Given X, When Y, Then Z"* rows verbatim where the guard is cited; resolver questions where the guard is inferred); **`[AI-SUGGESTED: AI-NNN | blocking]` markers on inferred lane assignments** → routed to resolver questions same as `missing-actor` disconnects (the resolver pipeline ingests these uniformly via the existing grammar — no schema changes); **`[AI-SUGGESTED: AI-NNN | non-blocking]` markers on inferred payloads** → routed to diagnostics as informational entries (the consultant may opt to address these but they do not gate `/requirements` sign-off); **inline-SVG swim-lane diagram + its Mermaid `flowchart TD` export source** → primarily for consultant comprehension (the SVG renders in-page; the Mermaid source is a copy-paste export adjunct, optionally rendered via mmdc or mermaid.live); neither is consumed by downstream agents (the YAML structured model is the machine-readable contract); **multi-process artefacts** (artefacts where `process_count > 1`) → flagged for `/requirements` as evidence the engagement covers ≥ 2 cross-functional workflows (the drafter should produce separate `§5 Task flows` entries per process and separate acceptance-criteria groups per process); **`[GAP-ACTOR-CONFLATION]` diagnostics** → flagged as blocking elicitation items (two distinct actor names that may or may not be the same actor must be resolved before `/requirements`); **SVG layout-warning diagnostics** (from `svg-overlap-check`) → flagged for the analyser-author / framework maintainer as evidence the inline-SVG geometry rules need refinement (not routed to design consumers); **manifest fingerprint** (the `<script type="application/json" id="swim-lane-process-mapping-meta">` block's `manifest_sha256`) → used by re-runs to detect drift between the analysis and the current manifest; not routed to design consumers.

**Inputs:** `analyse-inputs/SWIM-LANE-PROCESS-MAPPING/swim-lane-process-mapping.html`, `framework/assets/pattern-catalogue/` (for acceptance-criteria / consultant-question pattern entries when authored). The downstream `/requirements` drafter reads the HTML's converted-markdown sibling via the manifest, not this skill directly.

**Outputs:** Requirements-pipeline signal entries (process seeds for `§5`; handoff seeds for `§5 Task flows` *Decision points* / *Exception paths* and `§6` integration constraints; decision-branch guards as `§6` acceptance-criteria branches; actor-inventory entries for `§3` personas and `§2` domain model aggregates; data-store and external-system step nouns for `§7` and `§2` external-system aggregates), resolver-pipeline entries (every `consultant_follow_up: yes` disconnect as an `AI-NNN`-eligible question; `conflicting-source` disconnects as blocking resolver questions; inferred-actor / inferred-lane `[AI-SUGGESTED: AI-NNN | blocking]` markers as resolver questions), design-spec inventory entries (deferred — populated when the design-spec drafter is built), informational entries (silent concerns: `[GAP-ACTOR-CONFLATION]`, SVG layout warnings; manifest fingerprint).

**Used by:** `framework/agents/requirements-drafter.md` (when this artefact is re-ingested via `input/` and surfaces on the manifest — the drafter reads the converted-markdown sibling's YAML structured model as cross-functional-workflow scaffolding; specifically, processes become `§5 Task flows` seeds, handoffs become `§5 Task flows` *Decision points* / *Exception paths* + `§6` integration constraints, actor inventories become `§3 Target users` + `§2 Domain model` external-system seeds), `framework/agents/requirements-resolver.md` (when disconnect-register questions and inferred-node `AI-NNN` markers re-surface as resolver questions in the unified namespace; `conflicting-source` disconnects specifically are routed as blocking questions because they reflect real disagreements between the consultant's sources), and `framework/agents/design-spec-drafter/agent.md` (when authored — consumes step types, lane assignments, and handoff payloads as form-action / role-permission / integration-shape candidates). Also conceptually consumed by any future requirements-quality reviewer that consumes input-analyser output, and by `framework/skills/completeness-gap-pass.md` as a bijection target (every process → ≥ 1 `§5` entry; every handoff → ≥ 1 `§6` integration row).

**Used how:** Called when the swim-lane process mapping is present and the consultant has copied `analyse-inputs/SWIM-LANE-PROCESS-MAPPING/swim-lane-process-mapping.html` into `input/` for `/requirements` re-ingestion. Combines with other `map-*-to-ui` skills via the accrue-all pattern — each skill contributes complementary entries to the same downstream inventory. The swim-lane-process-mapping skill focuses on **cross-functional workflow signals + handoff disconnect signals + lane-ownership signals**, distinct from the journey-mapping skill (linear single-persona emotion signals), the thematic-analysis skill (cross-cutting pattern signals), the task-analysis skill (single-actor hierarchical decomposition signals), the opportunity-solution-trees skill (discovery-tree signals → opportunity-routing), the JTBD skill (motivation-shaped signals), and the glossary skill (vocabulary-health signals). Swim-lane process mapping never authors UI; it only surfaces structural workflow signals and the gaps in those workflows. The analyser's **primary** downstream consumer is the consultant (via the visible Disconnect Register driving the next elicitation conversation); the secondary consumer is the `/requirements` drafter (via re-ingestion). The Disconnect Register's analytical value compounds across pipelines: closing disconnects at this stage prevents fabricated integration requirements that would otherwise propagate silently through `/requirements` and into the eventual implementation.
