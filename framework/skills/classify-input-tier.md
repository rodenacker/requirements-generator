# classify-input-tier.md

**Purpose:** Map each file under `input/` to a readability tier. The tier determines how the input-handler treats the file: which are read directly as text, which are vision-described to a frozen sibling, which are rendered-then-described, which need MCP-backed conversion, which are recorded but unread. Used by the input-handler before conversion and manifest authoring.

**Inputs:**
- A list of file paths under `input/`. Dotfiles (paths whose name begins with `.`) and scratch files (paths whose name begins with `.`, used for the consultant's own private notes) are excluded by the input-handler before this skill is called.

**Outputs:** one row per input file with the fields `{ filename, original_path, extension, tier, kind }`. `tier ∈ { "Native-text", "Native-multimodal", "Vector-renderable", "Supported-via-MCP", "Unsupported" }`. `kind` defaults to `"primary"` for every file in `input/`; the `derived` value is reserved for files generated by other steps (the input-handler's own `*.converted.md` siblings carry `kind: "primary"` on their parent's row, not as separate rows).

**Used by:**
- `framework/agents/input-handler.md` — called immediately after enumeration, before preflight. Shared between `/requirements` and `/analyse-inputs`.

## Tier rubric

Resolve in order. The first step that produces a tier wins.

1. **Extension match — closed lists.** Match by file extension, case-insensitive.

    | Tier | Extensions | Read mechanism |
    |---|---|---|
    | `Native-multimodal` | `.png`, `.jpg`, `.jpeg`, `.gif`, `.webp` | The input-handler generates a frozen textual description sibling via `framework/skills/describe-visual-input.md` (Claude vision). Downstream consumers read the sibling, **not** the pixels — interpret-once. |
    | `Vector-renderable` | `.svg`, `.drawio`, `.vsdx` | The input-handler renders the vector to a raster via `framework/skills/render-visual-to-raster.md`, then describes the raster via `framework/skills/describe-visual-input.md`, writing a frozen description sibling consumers read. Requires an external renderer (gated by `framework/skills/preflight-cli.md`). |
    | `Supported-via-MCP` | `.docx`, `.xlsx`, `.pptx`, `.pdf` | Convert to a sibling `*.converted.md` via `framework/skills/convert-input-file.md`; the drafter reads the sibling, not the original. |

    These lists stay closed because each tier is bound to a specific read mechanism (vision input, vector-render-then-vision, markitdown MCP). New extensions land here only when the corresponding read path is actually available. **`Vector-renderable` is resolved here in step 1 before the content sniff** so that `.svg` / `.drawio` (which are XML text and would otherwise sniff as `Native-text`) are not mis-classified as directly-readable text.

2. **Content sniff — open default.** For every other file (including extensionless files), read the first 8192 bytes of `original_path` and apply this test, in order:

    - Empty file (0 bytes) → `Native-text`.
    - Buffer contains any `0x00` (NUL) byte → `Unsupported`. NUL bytes are the standard binary indicator (same heuristic git and `file(1)` use).
    - Strict UTF-8 decode succeeds (after stripping a leading UTF-8 / UTF-16 BOM if present) → `Native-text`.
    - Printable-ratio over the buffer (ASCII `0x20`–`0x7E` plus `\t \n \r`) is ≥ 0.95 → `Native-text`. Catches single-byte legacy encodings (Latin-1, Windows-1252) without forcing a decode guess.
    - Otherwise → `Unsupported`.

    Sniffing is **only** used to decide text-vs-binary for the open default. It never overrides an extension that already matched a closed list in step 1.

The drafter `Read`s `Native-text` rows directly regardless of extension; downstream behaviour is identical for a `.md`, a `LICENSE`, or a `Makefile` once classified as `Native-text`.

## Edge cases

- **No extension** — falls through to the content sniff. A `LICENSE`, `Dockerfile`, or `Makefile` whose bytes decode as text classifies as `Native-text`; a binary blob with no extension classifies as `Unsupported`.
- **Compound extension** (e.g. `report.tar.gz`, `chart.svg.png`) — match on the *last* extension only against the closed lists. `chart.svg.png` matches `Native-multimodal`. `report.tar.gz` does not match a closed list and falls through to the sniff (which will see binary bytes and return `Unsupported`).
- **Standalone images vs embedded images** — standalone `.png`/`.jpg`/etc. files in `input/` are `Native-multimodal`. Images embedded inside a `.docx` or `.pdf` are not separately registered; they are handled as part of the parent file's conversion (markitdown extracts captions or summaries inline) and recorded on the parent row's `conversions_applied` string.
- **Vector diagrams** — `.svg`, `.drawio`, and `.vsdx` are `Vector-renderable`. `.svg`/`.drawio` are XML text and `.vsdx` is a ZIP container (binary), but all three match the closed list in step 1, so the content sniff never runs for them — an `.svg` is therefore *not* mis-classified as `Native-text`, and a `.vsdx` is *not* mis-classified as `Unsupported`. The render-then-describe mechanism is gated by an external renderer; for `.svg`/`.vsdx`, if the renderer is unavailable the input-handler demotes these rows to `Unsupported` at conversion time. `.drawio` is the exception: when the `drawio` binary is absent or a render fails, the input-handler falls back to reading the mxGraph XML directly (`framework/skills/decode-drawio-xml.md`) rather than demoting — so a `.drawio` is only `Unsupported` if that fallback also fails. (None of this happens here — this skill only assigns the tier.)
- **`.converted.md` siblings** — these are produced by the input-handler itself for `Supported-via-MCP` files. They are not classified by this skill; they are referenced from the parent file's manifest row's `converted_sibling` field.
- **Encrypted or password-protected Office files** — classify as `Supported-via-MCP` based on extension; the conversion skill handles the runtime failure (markitdown errors out) and reclassifies the row to `Unsupported` with `conversions_applied: "failed — encrypted"`.
- **Text dump of binary data** (e.g. a Base64 payload) — classifies as `Native-text` because that is what the bytes are. The drafter sees text; the manifest's `sha256` still pins the original.

## Self-validation

- Every input file produces exactly one row. No file is dropped without classification.
- No row has `tier: null` or `tier: undefined`. Every file lands in one of the five tiers.
- The skill performs no `Write`. It performs at most one bounded prefix `Read` (≤ 8192 bytes) per file, and only for files that did not match a closed-list extension in step 1. Classification is deterministic: same bytes → same tier across runs.

## Anti-Patterns

- Do not use the content sniff to override an extension that already matched the `Native-multimodal`, `Vector-renderable`, or `Supported-via-MCP` closed list. A `.docx` whose first 8 KB look like text is still `Supported-via-MCP`, and a `.svg` (XML text) is still `Vector-renderable` — the closed lists are bound to read mechanisms, not to content shape.
- Do not extend the sniff to guess between `Supported-via-MCP` and `Native-text` (e.g. "this looks like markdown, treat it as `Native-text` even though it's a `.docx`"). The sniff's only job is text-vs-binary for the open default.
- Do not invent new tiers. The five-tier set (`Native-text`, `Native-multimodal`, `Vector-renderable`, `Supported-via-MCP`, `Unsupported`) is the contract between this skill, the manifest schema (`framework/skills/build-source-manifest.md`), the input-handler's conversion dispatch (`framework/agents/input-handler.md` step 5), and the downstream read-path resolution. A new tier requires coordinated edits across all of these. Note that consumers do **not** branch on tier to choose a read path — they follow the canonical Read-path resolution rule (read `converted_sibling` when non-null, else `original_path`), so a new tier that produces a sibling is invisible to them.
- Do not classify dotfiles or scratch files. The input-handler excludes them before calling this skill; classifying them would record private consultant notes in the manifest.
- Do not promote `Unsupported` to `Supported-via-MCP` because a future markitdown release might handle it. Add the extension to the closed list only when the conversion path is actually available.
- Do not read more than the prefix for the sniff. Reading the whole file pulls binary content into context on every classification; the 8 KB cap keeps the sniff cheap and bounded.
