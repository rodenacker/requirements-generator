# map-journey-mapping-from-inputs-to-ui.md

**Purpose:** Translate a current-state journey-mapping atlas (`analyse-inputs/JOURNEY-MAPPING/journey-mapping.html`) into signal-based inputs for the design-spec and requirements-drafting consumers. Unlike `map-task-flows-to-ui.md` (decomposition → affordances) or `map-ooux-to-ui.md` (objects → screens), Journey Mapping produces **signals about the user's current-of-the-world workflow** — phases, touchpoints, channels, emotions, pain points, backstage systems, opportunities, moments of truth — surfaced from raw consultant inputs, not direct UI affordances. The mapping is therefore: **phase backbones** (one per persona × scenario, 3–6 phases per backbone with verb-phrase labels and `[SRC: <filename>]` anchors) → routed to `§5 Task flows` outline seeds for the `/requirements` drafter (each phase becomes a candidate top-level task-flow step; the drafter normalises the verb phrasing and links into the `§4` user-stories backbone); **action lanes** (every verb-phrase user action with its source quote) → routed to functional-requirement seeds for `§6.2` (each action is a candidate `the system must allow X` clause; the drafter assigns `R-` IDs and merges); **touchpoint + channel lanes** → routed to screen / view inventory seeds in `§8 Source UI references` and to multi-platform requirement seeds in `§6.2` (a step with touchpoint `mobile app` plus channel `mobile` seeds a mobile-coverage requirement); **thought lanes** (verbatim quoted user thoughts) → routed to help / onboarding / confirmation / microcopy requirement seeds in `§6.4 UX requirements` (a quoted thought is the question the microcopy must answer at that step); **pain-point lanes** → routed to non-functional requirement seeds in `§6.1` (performance / error-handling / discoverability / recoverability NFRs, each anchored to a specific pain via `[SRC: <filename>]`); **backstage lanes** → routed to integration-requirement seeds in `§6.5 Integrations` (each backstage system named in the inputs becomes a candidate system-to-system contract); **opportunity lanes** (verb-led, solution-agnostic statements with parent-pain inheritance) → routed to user-story seeds for `§4 User goals & stories` (the drafter normalises the *"As a <persona>, I want to <opportunity-verb-clause> so that <pain-context>"* shape); **sentiment curves** (per-persona emotion score across phases, integer −2…+2) → routed to drafter-priority signals (low-sentiment phases get higher prioritisation in the `§9 Priorities` ordering; the drafter weights pain in low-sentiment phases above pain in neutral phases); **moments of truth** (phases flagged where sentiment drops ≥ 2 or stakes language present) → routed to reliability + observability requirement seeds in `§6.3 NFRs` (each moment of truth is a candidate P0 reliability annotation: "this touchpoint must not fail silently"; the drafter assigns SLO targets and merges); **`[GAP-NO-EVIDENCE]` diagnostics entries** → routed to consultant-interview prompts in the requirements-quality consumer (each gap is a question the consultant should answer before `/requirements` runs or before sign-off — *"the inputs don't evidence the user's thoughts during the `Investigation` phase; do you want to add interview material covering that phase, or accept the gap?"*); **the embedded `<script type="application/json" id="journey-mapping-meta">` block** → used by re-runs to detect drift between the atlas and the current manifest; not routed to design consumers.

**Inputs:** `analyse-inputs/JOURNEY-MAPPING/journey-mapping.html`, `framework/assets/pattern-catalogue/` (for task-flow / functional-requirement / NFR / integration / user-story pattern entries when authored).

**Outputs:** Design-spec inventory entries (screen / view inventory candidates, multi-platform-coverage flags), requirements-drafter signal entries (task-flow outline seeds for `§5`, functional-requirement seeds for `§6.2`, NFR seeds for `§6.1`/`§6.3`, integration-requirement seeds for `§6.5`, user-story seeds for `§4`, UX / microcopy seeds for `§6.4`, prioritisation signals for `§9`), informational entries (`[GAP-NO-EVIDENCE]` consultant-interview prompts, manifest fingerprint for drift detection).

**Used by:** `framework/agents/design-spec-drafter/agent.md` (when authored) and any future requirements-quality reviewer that consumes input-analyser output. Also conceptually consumed by the `/requirements` drafter when this artefact is re-ingested via `input/`: HTML classifies as `Markitdown-text` tier; the input-handler produces a `journey-mapping.converted.md` sibling via markitdown which preserves inline `[SRC: <filename>]` markers as plain text. The drafter reads the converted markdown's swim-lane cells as task-flow / functional-requirement / NFR / integration / user-story seeds; the drafter's `[SRC: C-NNN]` claim IDs cite the converted journey map, and the journey map's own `[SRC: <original-filename>]` markers within the cited content trace the audit trail end-to-end through to the original brief that justified the cell. The merger retains both citation layers in the final `requirements.md`.

**Used how:** Called when the journey-mapping atlas is present. Combines with other `map-*-to-ui` skills via the accrue-all pattern — each skill contributes complementary entries to the same downstream inventory. The journey-mapping skill focuses on **temporal-workflow signals, sentiment-derived prioritisation, and pain-to-opportunity bridging**; the thematic-analysis skill focuses on **cross-cutting-pattern signals and feature-cluster routing**; the OST skill focuses on **outcome-tree signals and bridge-to-requirement routing**; the glossary skill focuses on **vocabulary-health signals and label-source routing**; the jtbd skill focuses on **outcome-statement signals and job-to-screen routing**. Journey mapping never authors UI; it only surfaces signals other consumers anchor to. The analyser is **re-ingestible by `/requirements`** as a fresh source via markitdown: the drafter reads its swim-lane action cells as functional-requirement seeds (`§6.2`), its thought cells as microcopy seeds (`§6.4`), its pain cells as NFR seeds (`§6.1`/`§6.3`), its backstage cells as integration seeds (`§6.5`), its opportunity cells as user-story seeds (`§4`), and its sentiment curves as prioritisation signals (`§9`). The audit trail from the requirements draft back to the original brief is preserved through the dual-citation chain (drafter's `[SRC: C-NNN]` → journey map's `[SRC: <filename>]`).
