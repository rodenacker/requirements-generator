# map-task-analysis-from-inputs-to-ui.md

**Purpose:** Translate a Hierarchical Task Analysis of raw inputs (`analyse-inputs/TASK-ANALYSIS/task-analysis.html`) into signal-based inputs for the downstream requirements-drafting and design-spec pipelines. Unlike `map-task-flows-to-ui.md` (which maps the analyse-requirements-side task-flows artefact — HTA + TFD — into screen-flow inventory entries) or `map-ooux-to-ui.md` (which maps objects → screens), task-analysis of raw inputs produces **structural signals about goal decomposition, coordination logic, and per-terminal data nouns**, not direct UI affordances — analyses are extraction; design and consolidation are downstream concerns. The mapping is therefore: **YAML structured tree** (the `<pre><code class="language-yaml">` block inside `analyse-inputs/TASK-ANALYSIS/task-analysis.html`, carrying every node's `id` / `parent_id` / `label` / `is_operation` / `plan` / `information_required` / `sources` / `inferred` / `confidence`) → routed primarily to the `/requirements` drafter via re-ingestion through `input/` (the consultant copies the HTML into `input/`; input-handler classifies as `Supported-via-MCP`; markitdown converts to `.converted.md` preserving the `<pre><code>` block as fenced code; drafter reads the YAML structured tree as a bijection target for `framework/skills/completeness-gap-pass.md` — every terminal operation should map to ≥ 1 requirement; every requirement should map to ≥ 1 terminal or be flagged unattributed); **terminal operations** (every node with `is_operation: true`) → routed to candidate-requirement seeds for the `/requirements` drafter (the drafter normalises voice, assigns `R-` IDs, and merges into `§5 Task flows` *Steps* and `§6 Requirements`); **Plans (selection / iteration / concurrent)** → routed to acceptance-criteria branches for `/requirements` (selection Plans become *"Given X, When Y, Then Z"* branches; iteration Plans become *"the user may repeat …"* rules with the termination condition as the loop invariant; concurrent Plans surface as session / state-locking requirements in NFR or `§6` integration constraints; discretionary Plans become *"may be performed in any order"* notes); **information_required arrays** (every terminal's `{noun, direction, sources}` entries) → routed to `§7 Data entities` hints (the read/write nouns per terminal are precursors to data-model nodes; nouns appearing across ≥ 2 terminals are first-class entity candidates; nouns appearing with both `read` and `write` directions are mutable entities); **`[AI-SUGGESTED: AI-NNN | blocking]` markers** → routed to resolver questions in the `/requirements` resolver pipeline (each blocking gap allocates an `AI-NNN` identifier in the shared namespace with `framework/shared/general-rules.md`; the resolver ingests these uniformly via the existing grammar — no schema changes); **`[AI-SUGGESTED: AI-NNN | non-blocking]` markers** → routed to diagnostics as informational entries (the consultant may opt to address these but they do not gate `/requirements` sign-off); **silent-Plan branches** (every `[GAP-PLAN-SILENT]` entry in the diagnostics section) → routed to resolver questions same as blocking gaps; **trees that exceed depth 5 or the 2–10 sub-goal-per-parent caps** → flagged for design-spec consumers as candidates for cluster splitting (a 12-child sub-goal often corresponds to a screen that should split into two; a depth-6 sub-tree often indicates the goal should split into two independent goals); **multi-goal artefacts** (artefacts where `goal_count > 1`) → flagged for `/requirements` as evidence the engagement covers ≥ 2 independent user goals (the drafter should produce separate `§4` story groups and `§5` task-flow groups per goal); **manifest fingerprint** (the `<script type="application/json" id="task-analysis-meta">` block's `manifest_sha256`) → used by re-runs to detect drift between the analysis and the current manifest; not routed to design consumers.

**Inputs:** `analyse-inputs/TASK-ANALYSIS/task-analysis.html`, `framework/assets/pattern-catalogue/` (for acceptance-criteria / consultant-question pattern entries when authored). The downstream `/requirements` drafter reads the HTML's converted-markdown sibling via the manifest, not this skill directly.

**Outputs:** Requirements-pipeline signal entries (terminal-operation seeds for `§5` and `§6`; Plan-derived acceptance-criteria branches; information-requirement nouns for `§7`), resolver-pipeline entries (blocking gaps as `AI-NNN` questions; non-blocking gaps as informational entries), design-spec inventory entries (cluster-splitting flags for over-cap sub-goals; multi-goal flags for independent goal frames), informational entries (silent concerns, manifest fingerprint).

**Used by:** `framework/agents/requirements-drafter.md` (when this artefact is re-ingested via `input/` and surfaces on the manifest — the drafter reads the converted-markdown sibling's YAML structured tree as candidate-requirement scaffolding), `framework/agents/requirements-resolver.md` (when blocking gaps from the analysis re-surface as `AI-NNN` resolver questions in the unified namespace), and `framework/agents/design-spec-drafter/agent.md` (when authored — consumes terminal-operation labels as form-field / CTA candidates and `information_required` nouns as data-model seeds). Also conceptually consumed by any future requirements-quality reviewer that consumes input-analyser output.

**Used how:** Called when the task analysis is present and the consultant has copied `analyse-inputs/TASK-ANALYSIS/task-analysis.html` into `input/` for `/requirements` re-ingestion. Combines with other `map-*-to-ui` skills via the accrue-all pattern — each skill contributes complementary entries to the same downstream inventory. The task-analysis skill focuses on **structural-decomposition signals + per-terminal data signals + coordination-logic signals**, distinct from the journey-mapping skill (linear workflow signals per persona), the thematic-analysis skill (cross-cutting pattern signals), the opportunity-solution-trees skill (discovery-tree signals → opportunity-routing), and the glossary skill (vocabulary-health signals). Task analysis never authors UI; it only surfaces structural signals other consumers anchor to. The analyser is the only input-analyser whose **primary** downstream consumer is the `/requirements` drafter rather than a design author — its structured YAML is purpose-built for re-ingestion: terminal operations become candidate requirements (one-to-many bijection), Plans become acceptance criteria (selection → branching, iteration → loop, concurrent → state-lock), `information_required` arrays become `§7` data-model hints, and `[AI-SUGGESTED: AI-NNN | blocking]` markers flow into the resolver's existing question-grammar without modification.
