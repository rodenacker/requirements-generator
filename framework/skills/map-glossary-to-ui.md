# map-glossary-to-ui.md

**Purpose:** Translate a glossary analysis (`analyse-requirements/GLOSSARY/glossary.md`) into signal-based inputs for the design and requirements-quality pipelines. Unlike `map-task-flows-to-ui.md` (which maps decomposition → affordances) or `map-ooux-to-ui.md` (which maps objects → screens), Glossary produces **signals about domain-vocabulary health**, not direct UI affordances — analyses are extraction; design and enrichment are downstream concerns. The mapping is therefore: **undefined-term coverage signal** (count of "used without explicit definition" entries / total entries; per-section breakdown of where undefined terms appear most) → flagged for `/requirements` enrichment of `§2 Domain model`, `§3 Target users`, and `§7 Data entities` (high undefined-term counts mean the doc uses vocabulary it does not define — the consultant adds definitions, re-runs `/requirements` to refresh the merged doc, then re-runs `/analyse-requirement glossary` to verify the gap is closed); **acronym density** (count of tier-2 entries / total entries; ratio of expanded vs unexpanded acronyms) → flagged for screen-readability review and routed to in-app-glossary-pattern entries in the design system (high acronym density warns that the design system needs an affordance to disambiguate terms users may not recognise); **named-status-value catalogue** (every category=`status` entry) → routed to status-pill / badge / state-tag pattern entries in the design system, with the in-app label drawn verbatim from the glossary's definition body (the consultant's own document is the source of truth for status labels, not designer invention); **actor-role catalogue** (every category=`role` entry) → routed to role-switcher / permission-display patterns and to copy-tone style entries (different roles often warrant different tones — operator vs admin vs end-user); **action-verb catalogue** (every category=`action` entry) → routed to CTA-copy patterns (button labels, menu items, confirmation modals); **field-name catalogue** (every category=`field` entry, with `owning_entity` populated) → routed to form-field-pattern entries and to schema-anchor checks (the field names in the design must trace verbatim to glossary entries, not paraphrase them); **vocabulary-tier signal** (the active scope tier of the last run) → signal for design-spec complexity estimation — vocabulary-rich domains (tier 3 or 4 with > 50 entries) typically need more disambiguation in screen titles, list columns, and form labels.

**Inputs:** `analyse-requirements/GLOSSARY/glossary.md`, `framework/assets/pattern-catalogue/` (for in-app-glossary / status-pill / CTA-copy / form-field pattern entries).

**Outputs:** Requirements-quality signal entries (undefined-term coverage, definition-gap flags) for the requirements-quality consumer; pattern-catalogue trigger entries (status-pill labels, role-switcher entries, CTA copy, form-field labels) for the design-spec consumer; copy-tone style entries for the in-app-glossary / disambiguation surfaces.

**Used by:** `framework/agents/design-spec-drafter/agent.md` (when authored) and any future requirements-quality reviewer that consumes analyser output.

**Used how:** Called when the glossary analysis is present. Combines with other `map-*-to-ui` skills via the accrue-all pattern — each skill contributes complementary entries to the same downstream inventory. The glossary skill focuses on **vocabulary-health signals and label-source routing**; the task-flows skill focuses on **single-actor screen sequences**; the activity-diagram skill focuses on **multi-actor swimlane partitioning**; the five-whys skill focuses on **justification-health signals and policy/KPI pattern routing**. Glossary never authors UI; it only surfaces the vocabulary inventory other consumers anchor to.
