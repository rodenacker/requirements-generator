# map-affinity-mapping-from-inputs-to-ui.md

**Purpose:** Translate an inputs-side affinity map (`analyse-inputs/AFFINITY-MAPPING/affinity-map.html`) into signal-based inputs for the design and requirements-quality pipelines. Unlike `map-task-analysis-from-inputs-to-ui.md` (which maps decomposition → affordances) or `map-ooux-from-inputs-to-ui.md` (which maps objects → screens), affinity mapping produces **bottom-up clustered signals at the per-claim atomic level**, anti-anchored via a sub-agent-isolated Pass-2 re-cluster, with insight-statement labels that the downstream `/requirements` drafter can consume directly. The mapping is therefore: **L3 super-themes** (every entry in `super_themes[]`, with its insight-statement label and member cluster list) → routed to §1 vision-anchor seeds in the design-spec inventory (each super-theme becomes a candidate vision-anchor; the insight-statement label seeds the anchor's first sentence; the supporting cluster list anchors it to its evidence trail through the embedded JSON); **L2 clusters** (every entry in `clusters[]`, with its insight-statement label, member-note list, confidence distribution, and representative quote) → routed to candidate feature-area entries in the design-spec inventory (each cluster becomes a candidate feature area; the label seeds the area's headline; the representative quote seeds the area's elaboration; the `confidence_distribution.drifted` count flags the area for second-look review) AND to §3 acceptance-criteria seeds for the `/requirements` drafter (the drafter normalises voice, assigns `R-` IDs, and merges into §3; `[SRC: <filename>]` markers from the analysis coexist with the drafter's `[SRC: C-NNN]` claim IDs in the draft and are stripped by the merger); **orphans** (every entry in `orphans[]`, with its one-line "why no cluster" reason) → routed to §10 out-of-scope candidates for the `/requirements` drafter (each orphan is a single-stakeholder concern / edge case / future-scope hint the drafter should surface as a `[OUT-OF-SCOPE: domain-default]` line or as a resolver-pipeline `AI-NNN` question depending on its `reason`); **tensions** (every entry in `tensions[]`, with its `description` field naming the trade-off and its `sources` field naming the citing files) → routed to §6 trade-off-aware acceptance-criteria branches for the `/requirements` drafter (each tension is a candidate branch point in §6 — the methodology already named the two clusters in conflict, the drafter expresses the trade-off as a conditional acceptance-criterion or as a resolver-pipeline `AI-NNN` question); **drifted notes** (every entry in `notes[]` where `confidence == "drifted"`, with its `jaccard` value and `drifted_from_pass2_label`) → flagged for design-spec consumers as lower-confidence cluster placements (a drifted note's cluster assignment was contested by the anti-anchoring control; the consultant's review is the canonical conflict-resolution surface); **manifest fingerprint** (the `manifest_sha256` field in both the `<script type="application/json" id="affinity-map-meta">` block and the `<pre><code class="language-json" id="affinity-map-body">` block) → used by re-runs to detect drift between the analysis and the current manifest; not routed to design consumers.

**Inputs:** `analyse-inputs/AFFINITY-MAPPING/affinity-map.html` (specifically the embedded `<pre><code class="language-json" id="affinity-map-body">` block — survives markitdown round-trip and is the load-bearing machine-readable contract), `assets/taxonomy-screens.md` (for feature-area / vision-anchor / out-of-scope / trade-off pattern entries when authored).

**Outputs:** Design-spec inventory entries (vision-anchor seeds from super-themes, feature-area candidates from clusters, drifted-cluster flags), requirements-quality signal entries (acceptance-criteria seeds from clusters, out-of-scope candidates from orphans, trade-off-aware branch seeds from tensions), informational entries (manifest fingerprint, recommended consultant follow-up questions from diagnostics).

**Used by:** `framework/agents/design-spec-drafter/agent.md` (when authored) and any future requirements-quality reviewer that consumes input-analyser output. Also conceptually consumed by the `/requirements` drafter when this artefact is re-ingested via `input/` (drafter reads the embedded JSON body block's `super_themes[]` / `clusters[]` / `orphans[]` / `tensions[]` arrays as candidate-requirement / vision-anchor / out-of-scope / trade-off seeds, with the analysis's `[SRC: <filename>]` markers superseded by the drafter's `[SRC: C-NNN]` claim IDs in the draft).

**Used how:** Called when the affinity map is present. Combines with other `map-*-from-inputs-to-ui` skills via the accrue-all pattern — each skill contributes complementary entries to the same downstream inventory. The affinity-mapping skill focuses on **bottom-up clustered-claim signals and vision-anchor / trade-off routing**; the thematic-analysis skill focuses on **deductive-frame coverage signals and feature-cluster routing**; the ooux skill focuses on **object-structure signals and screen-taxonomy routing**; the swim-lane skill focuses on **handoff / disconnect signals and process-mapping routing**. Affinity mapping never authors UI; it only surfaces signals other consumers anchor to. The analyser is fully **re-ingestible by `/requirements`**: dropping `analyse-inputs/AFFINITY-MAPPING/affinity-map.html` into `input/` classifies it as `Supported-via-MCP` (the input-handler runs markitdown HTML→MD); the converted `.converted.md` carries the JSON body block as a fenced ```json code block, the Mermaid `mindmap` and `flowchart TD` blocks as fenced code blocks, and the `[SRC: <filename>]` markers as plain text. The drafter reads them as candidate-requirement / vision-anchor / out-of-scope / trade-off seeds; the merger strips both the analysis's `[SRC: <filename>]` markers and the drafter's `[SRC: C-NNN]` markers when producing the final `requirements/requirements.md`. Drifted-note flags surface in the design-spec consumer; the `/requirements` drafter does not normally re-weight by Jaccard value — it treats every clustered note as a candidate seed regardless of confidence (the drift information is metadata for human review, not a quantitative weighting input for the drafter).
